# V1.3 Release Hardening Implementation Plan

> **中文说明：** 本实施计划生成检索质量、重排延迟、Milvus 升级恢复、供应链风险和稀疏迁移运维的可重复发布证据。

> **For agentic workers:** REQUIRED SUB-SKILL: Use $superpower-subagents (recommended) or $superpower-executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking via update_plan.

**Goal:** Produce repeatable release evidence for retrieval quality, rerank latency, Milvus upgrade recovery, supply-chain risk, and Sparse Migration operations before publishing V1.3.

**Architecture:** Keep release hardening inside existing ownership boundaries: Worker manages model lifecycle, PowerShell scripts orchestrate benchmark/rehearsal/scanning, and the knowledge-base Web page operates the API-owned migration state machine. Generated evidence is written under `artifacts/`; stable fixtures, scripts, tests, and runbooks are committed.

**Tech Stack:** Python/FastAPI/pytest, sentence-transformers/PyTorch, Spring Boot/JPA, React/Vite/Vitest/Playwright, Docker Compose, PowerShell, Milvus 2.5.4, pip-audit, Syft, Trivy.

## Self-Review

- The plan covers every approved design section and keeps generated environment evidence out of commits.
- State and acceptance decisions remain server-owned; the Web panel cannot force an invalid transition.
- Reranker selection is evidence-driven and preserves the current model unless the smaller candidate passes quality and latency gates.
- Backup precedes upgrade, restore uses an isolated project, and Rollback is tested after Cutover.
- Documentation synchronization and `dev/` branch naming are explicit delivery requirements.
