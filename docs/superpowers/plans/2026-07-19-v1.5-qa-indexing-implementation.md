# V1.5 QA Indexing Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use $superpower-executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking via update_plan.

**Goal:** Add bounded QA candidate generation through the existing API LLM, persist QA chunks with provenance, and make `QA_ASSISTED`/`COMBINED` retrieve source context without changing classic defaults.

**Architecture:** The Worker remains responsible for ingest parsing, chunking, embedding, and Milvus writes. It calls an authenticated API internal endpoint for batched QA candidates; the API reuses `LlmClient.chat` and returns validated candidates. QA chunks are persisted with `source_chunk_id` metadata and indexed beside normal searchable entries; failures fall back to original chunks.

**Tech Stack:** Spring Boot/Jackson/WebClient, PostgreSQL chunk metadata, Python/httpx, Milvus, JUnit/Mockito, pytest.
