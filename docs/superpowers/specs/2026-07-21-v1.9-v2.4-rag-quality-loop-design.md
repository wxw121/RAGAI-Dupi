# V1.9-V2.4 RAG Quality Loop Design

Status: Recommended defaults selected by the user. This pass intentionally avoids Docker Compose, real browser E2E, and true load testing to protect local machine resources.

## Goal

Turn the V1.6b-V1.8 evaluation foundation into a broader RAG quality loop without adding heavyweight persistence or release-environment dependencies in this slice.

## Version Scope

- V1.9 Release quality closure: expose release readiness evidence, blocker counts, artifact keys, and required quality evidence from each evaluation run.
- V2.0 Real query feedback loop: convert failed or degraded evaluation signals into candidate feedback items that can seed future real-query review queues.
- V2.1 Retrieval experiment platform: summarize experiment dimensions from each run, including observed TopK values, profiles, retrieval modes, evaluation count, and rerank evidence.
- V2.2 Answer quality layer: add grounded-answer proxy metrics using citation eligibility, citation pass rate, and hallucination-risk counts.
- V2.3 Online quality observability: summarize fallback rate, no-answer correctness, latency P50/P95, and degraded profile count.
- V2.4 Data/index quality governance: summarize expected source coverage, missing source count, multi-document coverage, ambiguity coverage, and embedding dimension evidence.

## Architecture

Use the existing RagEvalRun.metrics JSON object as the integration surface. This keeps the current tables and migrations stable while allowing the web dashboard, benchmark artifacts, and release documents to consume the same versioned maps.

The API computes six additional maps during evaluation:

- releaseReadiness
- realQueryFeedback
- experimentMatrix
- answerQuality
- onlineObservability
- dataIndexGovernance

The web panel renders these as compact dashboard cards alongside the existing release gate, category summaries, profile comparisons, and diagnostic drilldown.

## Boundaries

- No new database tables in this pass.
- No background jobs in this pass.
- No automatic ingestion of production chat logs in this pass; V2.0 is represented as a safe candidate queue derived from failed/degraded eval signals.
- No true latency/load experiments in this pass; V2.3 uses evaluation-run latency and fallback evidence.
- Heavy validation is deferred to post-V2.4: browser E2E, Docker Compose rehearsal, and production-like load tests.

## Evidence Model

Each map includes a version field so artifacts can be interpreted even if later versions add fields. The quality loop is intentionally additive: existing passRate, categorySummaries, profileSummaries, profileComparisons, and releaseGate remain compatible.

## Definition of Done

- API metrics include all six versioned maps.
- API tests prove the maps are produced from a multi-profile evaluation run.
- Web types and dashboard render the six maps.
- Web component tests prove the dashboard labels render.
- README, progress, todo, design, and implementation docs describe the V1.9-V2.4 slice.
- Verification uses focused Maven, focused Vitest, TypeScript/build if feasible, Pester, and git diff hygiene; resource-heavy gates remain documented as deferred.
