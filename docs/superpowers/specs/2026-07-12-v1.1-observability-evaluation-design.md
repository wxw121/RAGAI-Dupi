# V1.1 Observability And Evaluation Design

> **中文说明：** 本设计通过真实浏览器端到端门禁、摄取与索引诊断、轻量 RAG 评估闭环以及上传/运维可见性，让现有 V1 系统更易测试、诊断和运营。

## Summary

V1.1 focuses on making the existing V1 system testable, diagnosable, and easier to operate. The target is not a broad platform rewrite. The target is four closed loops:

- real browser E2E gate for login and core UI flows;
- ingest/index diagnosis so failed or stalled documents explain what happened and what to do next;
- RAG evaluation so retrieval quality can be checked from scripts and from the product surface;
- P1 upload governance and ops alerts as visibility first, without adding new hard quota blocking in this version.

Current API-level E2E scripts and RAG scripts remain useful, but they do not prove that a user can operate the product in a browser. Current ingest and alert data exists, but too much of it is hidden in raw fields or scattered panels. V1.1 turns those capabilities into explicit gates and visible operator feedback.

## Approaches Considered

### Approach A: CI-only quality gates

Add Playwright and RAG scripts only, with no product UI changes.

Pros:
- Lowest code surface.
- Fast to add and easy to run in CI.

Cons:
- Does not help admins diagnose user-facing ingest failures.
- RAG evaluation remains invisible to non-engineers.
- Upload governance and alerts remain fragmented.

### Approach B: Full observability subsystem

Add persisted evaluation runs, alert rules, notification channels, upload quotas, and a full ops dashboard.

Pros:
- Long-term scalable direction.
- Clear audit trail for evaluations and alerts.

Cons:
- Too large for a small version.
- Introduces new schema and alerting semantics before the product has enough operating history.
- Risks delaying the P0 browser gate and ingest diagnosis.

### Approach C: Thin closure layer on existing flows

Add Playwright browser E2E, enrich existing ingest job responses with computed diagnosis, add a lightweight RAG evaluation tab that reuses `/retrieve`, and expose current governance thresholds plus actionable ops alerts.

Pros:
- Uses current architecture and existing services.
- Makes the most painful failure modes visible immediately.
- Gives both engineers and admins a repeatable quality check.
- Keeps P1 upload governance as visibility and alerting, not a new policy engine.

Cons:
- Evaluation reports are lightweight and not yet persisted historically.
- Upload quotas remain future work.
- Alert delivery is still in-product/API only, not webhook/email/IM.

Chosen approach: Approach C.

## Scope

### P0: Real Browser E2E Gate

Add Playwright under `services/web` and a root PowerShell wrapper:

- Login through the real page at `http://localhost:8080`.
- Use cookie + CSRF behavior exactly as the product uses it.
- Create a knowledge base through the UI.
- Navigate to the KB detail page.
- Verify document, chat, ops audit, accounts, and roles pages render without console errors.
- Upload flow can be included when the local stack is configured and stable; otherwise the first gate should still prove authenticated navigation and form flows.

Credentials are provided by environment variables:

- `E2E_ADMIN_USERNAME`
- `E2E_ADMIN_PASSWORD`
- `E2E_BASE_URL`, default `http://localhost:8080`

The gate fails with a clear message if credentials are missing. It must not rely on local open mode or bypass authentication.

### P0: Ingest/Index Diagnosis Closure

Extend existing ingest job responses with a computed diagnosis object. No schema migration is required.

Diagnosis fields:

- `severity`: `info`, `warning`, or `error`;
- `summary`: one short operator-facing explanation;
- `nextAction`: concrete action, such as wait, retry, reindex, check embedding config, or inspect storage/worker;
- `retryable`: whether manual retry is appropriate;
- `stalled`: whether the job has not updated for a practical threshold;
- `ageSeconds` and `lastUpdatedSeconds`.

The diagnosis is computed from existing fields:

- document status;
- job status;
- job stage;
- retry count;
- error message;
- timestamps.

UI changes:

- Document table shows the latest job state and a compact diagnosis line.
- Index maintenance table shows diagnosis, not only raw error text.
- Failed/dead-letter jobs expose retry as before, but now explain whether retry is likely useful.
- Stalled pending/processing jobs are called out without waiting for a hard failure.

### P0: RAG Evaluation Closure

Keep the existing `examples/rag-eval-cases.json` and `scripts/rag-regression-eval.ps1`, but make evaluation visible and reusable:

- Add a `RAG 评估` tab in KB detail.
- The page runs the same style of checks against the current KB using `/retrieve`.
- Show per-case pass/fail, hit count, expected file, matched token, retrieval mode, fallback reason, and embedding dimensions.
- Add a frontend API wrapper for `/retrieve` and shared eval case/result types.
- Update the script output so each case has enough detail to diagnose failures from the generated JSON report.

This version does not persist evaluation history. The latest page run is enough for operator feedback, and the script report is enough for gate evidence.

### P1: Upload Governance And Ops Alerts

This version exposes current operating guardrails and adds useful in-product alerts. It does not add a new hard quota engine.

Add metadata for:

- upload rate limit enabled/requests/window;
- ingest queue max pending jobs;
- ingest recovery max attempts;
- audit alert window and failed threshold;
- multipart max file size if available.

UI changes:

- Upload area shows supported file types always.
- If the current caller can load ops metadata, the upload area also shows the configured rate-limit and queue hints.
- Ops audit page shows current guardrail settings and active alerts.
- Alert list includes audit failure spike plus ingest dead-letter/failed jobs and vector cleanup failures where existing repositories can support counts.

Future quota enforcement remains out of scope for this version.

## Product Behavior

### Browser Gate

The gate is run by:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/e2e-browser-gate.ps1
```

It starts from a clean browser context, logs in, and fails on:

- missing credentials;
- login error;
- navigation timeout;
- visible error toast on the checked flows;
- uncaught page exception;
- browser console error from application code.

### Ingest Diagnosis

Admins should be able to answer three questions from the document page:

1. What is the document doing now?
2. If it failed, why did it fail?
3. What should I do next?

The UI should prefer short summaries over raw stack-style messages, while still showing the raw error when useful.

### RAG Evaluation

The evaluation page is intentionally simple:

- select or use built-in cases;
- click run;
- see pass/fail table;
- inspect retrieval diagnostics.

The initial built-in cases mirror `examples/rag-eval-cases.json`. Custom persisted case management is not part of V1.1.

### Ops Alerts

Alerts remain pull-based in V1.1. The product exposes them in the ops page and API. External notification integrations are deferred.

## Data And API Design

### Ingest Diagnosis

Add DTOs:

- `IngestDiagnosisResponse`
- optional fields on `IngestJobResponse`:
  - `documentFileName`
  - `documentStatus`
  - `diagnosis`

Existing endpoints remain:

- `GET /api/v1/knowledge-bases/{kbId}/ingest-jobs`
- `GET /api/v1/knowledge-bases/{kbId}/documents/{docId}/ingest-job`

### Retrieval API Client

No backend endpoint is required for the first UI evaluation pass. The web client calls:

- `POST /api/v1/knowledge-bases/{kbId}/retrieve`

Add frontend types for:

- `RetrieveRequest`
- `RetrieveResponse`
- `RetrievalHit`
- `RagEvalCase`
- `RagEvalResult`

### Ops Metadata

Extend `OpsMetadataResponse` with:

- `guardrails`

Suggested shape:

```json
{
  "guardrails": {
    "uploadRateLimit": {
    "enabled": true,
    "requests": 20,
    "windowSeconds": 60
    },
    "ingestQueue": {
      "maxPendingJobs": 200,
      "maxRecoveryAttempts": 3
    },
    "audit": {
      "alertWindowMinutes": 30,
      "alertFailedThreshold": 10
    }
  }
}
```

### Ops Alerts

Keep:

- `GET /api/v1/ops/audit-alerts`

Extend returned alerts with additional codes where practical:

- `AUDIT_FAILED_SPIKE`
- `INGEST_FAILURES_OPEN`
- `VECTOR_CLEANUP_FAILURES_OPEN`

## Testing

### Backend

Add focused tests for:

- ingest diagnosis computed from completed, failed, dead-letter, and stale jobs;
- ops metadata returns configured guardrail settings;
- ops alerts include ingest/vector cleanup summaries when repositories report open failures.

### Frontend

Add Vitest coverage for:

- retrieve API wrapper;
- RAG eval case runner pass/fail logic;
- ingest diagnosis rendering;
- operations metadata rendering;
- Playwright config sanity where practical.

### Browser E2E

Add Playwright tests for:

- login;
- KB create form;
- authenticated navigation;
- KB detail document/chat/eval tabs;
- ops pages render.

### Existing Gates

Run before claiming completion:

- API focused tests, then `mvn verify` if runtime allows;
- Web `npm test` and `npm run build`;
- Worker tests if ingest-related behavior is touched;
- `scripts/e2e-browser-gate.ps1` against the running local app when credentials and services are available;
- existing API-level E2E/RAG scripts when full stack credentials are configured.

## Out Of Scope

- SSO/OIDC.
- Webhook, email, or IM alert delivery.
- Hard upload quota enforcement per tenant/user.
- Persisted RAG evaluation history.
- Custom evaluation-case CRUD.
- K8s/Helm.
- Rewriting the ingest pipeline.

## Open Risks

- Full browser E2E depends on a running local stack and a real admin account. The gate must fail clearly when prerequisites are missing.
- RAG evaluation quality depends on current KB contents. Built-in sample cases are stable only when the sample document has been uploaded or when the user evaluates an equivalent KB.
- Some project text renders mojibake in PowerShell output, but source files are treated as UTF-8. Edits should preserve UTF-8.
