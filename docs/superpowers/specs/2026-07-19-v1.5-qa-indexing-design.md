# V1.5 QA Indexing Design

## Summary

Add the P1 QA-assisted indexing path without changing the existing classic or parent-child defaults. During ingest, the Worker sends source chunks to an internal API endpoint. The API reuses the configured `LlmClient` to generate validated question/answer candidates. The Worker turns candidates into QA chunks, stores them with source provenance, and indexes them beside the normal retrieval entries.

## Goals

- Generate a small, bounded set of candidate questions and answers for each source chunk.
- Preserve provenance from QA chunk to source chunk, document, and parent context.
- Make `QA_ASSISTED` retrieve both QA entries and original entries.
- Make `COMBINED` retrieve parent-child and QA entries while returning source context to the caller.
- Keep QA generation failure non-fatal: original chunks remain searchable.
- Keep `CLASSIC` and existing `VECTOR/HYBRID/RERANK` behavior unchanged.

## Non-goals

- No automatic default profile switch.
- No provider-specific LLM configuration in the Worker.
- No answer generation at query time; QA entries are retrieval shortcuts only.
- No score-weight tuning in this P1 batch.

## Chosen Architecture

### API QA generation endpoint

Add an internal endpoint:

`POST /api/v1/internal/knowledge-bases/{kbId}/qa-candidates`

Request:

```json
{
  "docId": "uuid",
  "sources": [
    {"chunkId": "uuid", "content": "...", "metadata": {}}
  ]
}
```

Response:

```json
{
  "candidates": [
    {
      "sourceChunkId": "uuid",
      "question": "...",
      "answer": "..."
    }
  ]
}
```

The API validates that the requested document belongs to the knowledge base, sends a strict JSON prompt through the existing `LlmClient.chat`, parses and validates candidates, removes duplicate questions, and caps the number of candidates per source. Source chunks are not yet persisted when QA generation runs, so their IDs/content are trusted only across the authenticated Worker/internal-key boundary.

### Worker ingest flow

- `CLASSIC`: unchanged.
- `PARENT_CHILD`: unchanged from P0; parents are persisted and children are indexed.
- `QA_ASSISTED`: normal chunks are persisted/indexed; normal chunks are sent as QA sources; generated QA chunks are persisted/indexed.
- `COMBINED`: parents and children are generated as in P0; parents are sent as QA sources; generated QA chunks are persisted/indexed beside children.

Each QA chunk uses content shaped as `Question: ...\nAnswer: ...` and metadata:

```json
{
  "chunk_role": "qa",
  "source_chunk_id": "uuid",
  "qa_question": "...",
  "qa_answer": "...",
  "qa_source_kind": "original_chunk"
}
```

The QA chunk receives its own UUID and chunk index. The original source chunk remains authoritative for citations and answer context.

### Failure and retry behavior

If the internal call, LLM response, JSON parsing, or candidate validation fails, the Worker logs the failure and completes ingest with the original chunks. No malformed QA chunk is persisted or indexed. Reindex/retry runs QA generation again, so a transient provider failure can recover without a separate migration.

### Retrieval behavior

For `QA_ASSISTED` and `COMBINED`, a hit whose metadata contains `source_chunk_id` is expanded to the source chunk. The response keeps QA provenance fields:

- `matched_qa_chunk_id`
- `qa_source_chunk_id`
- `qa_question`
- `qa_answer`
- `profile`

For `COMBINED`, parent expansion is applied after QA source resolution when the resolved source is a child. The returned content is always the source/parent context, while the metadata identifies the QA or child entry that caused the match. Cross-KB and cross-document source references are rejected.

Hybrid BM25 continues to exclude parent-only chunks as direct entries, but QA chunks remain eligible entries.

## Testing

- API: candidate prompt response parsing, validation/deduplication/capping, malformed response failure, endpoint delegation and KB/document boundary.
- Worker: QA candidate to `TextChunk` conversion, provenance metadata, QA-assisted and combined source selection, non-fatal generation fallback, indexing only searchable entries.
- Retrieval: QA hit expansion, combined QA-plus-parent expansion, provenance preservation, cross-document rejection.
- Existing classic/parent-child/hybrid tests remain green.

## Rollout Gate

Keep the default profile as `CLASSIC`. Enable `QA_ASSISTED` or `COMBINED` per knowledge base only after evaluation on an index physically built for that candidate profile shows no regression in hit rate and citation quality. The current multi-profile run compares retrieval behavior over one active physical index; profile-isolated indexes (or a filterable superset) and an automatic default-enable policy remain P2.
