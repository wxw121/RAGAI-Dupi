# V1.5 P2 Profile-Isolated Evaluation and Quality Gate Design

> **中文说明：** 本设计确保不同检索配置基于等价源数据进行比较，并要求非 `CLASSIC` 配置在命中率与引用质量不退化后才能成为知识库默认配置。

## Summary

P2 makes retrieval profile comparisons valid across equivalent source data and prevents a non-classic profile from becoming a knowledge base default until it proves that hit rate and citation quality do not regress against `CLASSIC`.

The chosen architecture builds one filterable Milvus superset for every document. It preserves the existing VECTOR/HYBRID/RERANK execution modes, keeps `CLASSIC` as the initial default, adds deterministic weighted fusion for `COMBINED`, and stores revision-bound gate decisions in RAG evaluation runs.

## Goals

- Build all four retrieval profiles from the same document version during one ingest.
- Filter vector and BM25 candidates by profile before ranking.
- Compare `CLASSIC` and candidate profiles against equivalent indexed source data.
- Fuse Combined child and QA routes with explicit, diagnosable weights.
- Reject non-classic default profile changes unless a current evaluation gate passes.
- Keep an old classic collection readable while existing knowledge bases rebuild the v2 profile index.
- Preserve non-fatal QA generation fallback and existing retrieval execution modes.

## Non-Goals

- Automatically change a knowledge base default profile without a user action.
- Add per-knowledge-base score tuning controls to the Web UI.
- Replace existing VECTOR/HYBRID/RERANK execution modes.
- Provide zero-cost indexing; the user accepted higher embedding, QA generation, and Milvus storage cost in exchange for valid A/B comparison and instant profile switching.
- Remove the legacy Milvus collection before every knowledge base has completed the v2 transition.

## Chosen Architecture

### Filterable Superset

Create a new Milvus collection named `${MILVUS_COLLECTION}_profiles_v2`. Do not attempt to add scalar fields to an existing collection in place.

Each indexed record contains the existing fields plus:

- `entry_kind`: `original`, `child`, or `qa`
- `profile_classic`: boolean
- `profile_parent_child`: boolean
- `profile_qa_assisted`: boolean
- `profile_combined`: boolean

PostgreSQL chunk metadata stores the same entry kind and a `profile_scope` list. Parent chunks remain PostgreSQL-only and are never direct vector or BM25 entries.

One ingest produces:

| Entry | Persisted | Vector indexed | Profile scope |
|---|---:|---:|---|
| Original chunks | yes | yes | `classic`, `qa-assisted` |
| Parent chunks | yes | no | `parent-child`, `combined` |
| Child chunks | yes | yes | `parent-child`, `combined` |
| QA from original chunks | yes | yes | `qa-assisted` |
| QA from parent chunks | yes | yes | `combined` |

Original chunks are reused by classic and QA-assisted. Child chunks are reused by parent-child and combined. QA entries remain separate because QA-assisted and combined use different source semantics.

### Profile Filtering

Profile filtering occurs before ranking:

- `CLASSIC`: `profile_classic == true`
- `PARENT_CHILD`: `profile_parent_child == true AND entry_kind == "child"`
- `QA_ASSISTED`: `profile_qa_assisted == true`
- `COMBINED` child route: `profile_combined == true AND entry_kind == "child"`
- `COMBINED` QA route: `profile_combined == true AND entry_kind == "qa"`

The Worker applies the equivalent `profile_scope` and `entry_kind` filters before BM25 scoring. Request-level profile overrides therefore select an isolated logical index rather than replaying different expansion behavior over one unfiltered physical index.

### Combined Weighted Fusion

Combined retrieves child and QA routes independently so one route cannot consume the other route's candidate budget.

Use weighted reciprocal rank fusion:

`routeScore = routeWeight / (rrfK + rank + 1)`

Defaults:

- child weight: `1.0`
- QA weight: `0.8`
- RRF K: `60`

Configuration keys are shared by API and Worker through environment-backed properties:

- `COMBINED_CHILD_WEIGHT`
- `COMBINED_QA_WEIGHT`
- `RRF_K`

All weights must be positive and `RRF_K` must be at least 1. Vector mode fuses child-vector and QA-vector. Hybrid mode fuses child-vector, child-BM25, QA-vector, and QA-BM25, using the child or QA weight for both execution routes. Optional reranking runs after fusion. Retrieval diagnostics include route counts, effective weights, and RRF K.

After QA source and parent expansion, final hits are deduplicated by final chunk ID and the highest fused score is retained.

## Index Version and Revision

Flyway V10 adds:

- `documents.index_schema_version INTEGER NOT NULL DEFAULT 1`
- `knowledge_bases.index_revision BIGINT NOT NULL DEFAULT 0`
- `rag_eval_runs.index_revision BIGINT`
- `rag_eval_runs.gate_summary JSONB NOT NULL DEFAULT '{}'::jsonb`
- `rag_eval_run_results.hit_passed BOOLEAN NOT NULL DEFAULT FALSE`
- `rag_eval_run_results.citation_eligible BOOLEAN NOT NULL DEFAULT FALSE`
- `rag_eval_run_results.citation_passed BOOLEAN NOT NULL DEFAULT FALSE`

The target document index schema version is `2`.

`index_revision` changes whenever indexed knowledge changes:

- a document ingest completes and its chunks replace the previous version;
- a document is deleted;
- a knowledge base reindex starts;
- an operation otherwise invalidates the current physical index.

An evaluation run captures the knowledge base revision before executing cases. A gate decision is current only when its run revision equals the knowledge base revision at profile-change time.

The knowledge base profile index is ready when every active document is indexed successfully with schema version 2. An empty knowledge base is not gate-ready.

## Upgrade and Cutover

The legacy and v2 collections coexist during rollout.

1. Before a knowledge base is v2-ready, only `CLASSIC` can be its default profile and classic retrieval reads the legacy collection.
2. A v2 reindex does not delete legacy vectors at the beginning.
3. While the knowledge base is not v2-ready, ingest writes classic original entries to the legacy collection and the full superset to the v2 collection.
4. Each successful callback marks its document schema version as 2.
5. When all active documents reach version 2, retrieval switches to the v2 collection for every profile.
6. Legacy vectors for that knowledge base are deleted after cutover. A failed delete is recorded through the existing vector-cleanup compensation mechanism with an explicit legacy-collection target.

Reindex resets affected document schema versions before enqueueing jobs. New uploads during a partial migration remain visible to classic because the Worker dual-writes their original chunks to the legacy collection.

## Evaluation Metrics

Each evaluation result records three independent outcomes:

- `hitPassed`: returned hit count satisfies the case's `minHits`.
- `citationEligible`: the case has a nonblank `expectedFileName`.
- `citationPassed`: an eligible case returned the expected file.

The existing overall `passed` result continues to include hit, citation, and expected-token requirements.

For every profile in a completed run, compute:

- `hitRate = hitPassedCount / caseCount`
- `citationRate = citationPassedCount / citationEligibleCount`
- `casePassRate = passedCount / caseCount`

`casePassRate` is diagnostic and does not control the default-enable gate.

## Gate Decision

A gate comparison requires:

- the profile index is v2-ready;
- the run includes `CLASSIC` and the candidate profile;
- the run is `COMPLETED`;
- the run revision matches the current knowledge base revision;
- at least 3 cases exist for each compared profile;
- at least 1 citation-eligible case exists for each compared profile.

A candidate passes only when:

- candidate hit rate is greater than or equal to classic hit rate; and
- candidate citation rate is greater than or equal to classic citation rate.

Gate statuses are:

- `PASSED`
- `BLOCKED`
- `INSUFFICIENT_DATA`
- `STALE`
- `INDEX_NOT_READY`
- `NOT_EVALUATED`

The run's `gate_summary` stores baseline metrics, candidate metrics, rate deltas, status, and machine-readable reasons for every non-classic profile included in the run.

## Default Profile Enforcement

New knowledge bases must start with `CLASSIC`. A create request that specifies another default is rejected.

Changing a default profile follows these rules:

- switching to `CLASSIC` is always allowed;
- switching to a non-classic profile requires `MAINTENANCE` permission;
- the API loads the latest completed run containing classic and the target profile;
- the stored gate must be `PASSED` and current for the knowledge base revision;
- a successful switch updates only the default profile because the superset is already built;
- a failed switch returns HTTP 409 with a stable code and a specific reason.

The system never switches a default profile automatically.

## API and Web Surface

Knowledge base responses add:

- `indexSchemaVersion`
- `profileIndexReady`
- `indexRevision`
- latest gate status for each non-classic profile

RAG evaluation run responses add:

- `indexRevision`
- per-profile metrics
- per-candidate gate decisions and deltas

The knowledge base settings panel displays v2 readiness and the latest gate status for each candidate profile. The server remains authoritative; disabled or stale client state cannot bypass the gate.

The RAG evaluation panel displays hit rate, citation rate, case pass rate, deltas from classic, and the gate status/reason. It does not expose score-weight tuning controls.

## Failure Handling

- QA generation failure remains non-fatal. Original, parent, and child entries still complete ingestion; QA-dependent profiles must prove quality through the gate.
- A non-classic request against a knowledge base whose v2 index is not ready returns a conflict error instead of silently searching the wrong index.
- A Combined route with zero hits contributes nothing and the other route still returns results.
- Milvus-wide failure preserves the existing local text fallback behavior, with PostgreSQL corpus filtered by requested profile.
- A revision change during evaluation marks the resulting gate stale and prevents profile activation.
- Legacy cleanup failure does not roll back successful v2 cutover; it enters compensation and remains observable.

## Security and Trust Boundaries

- Existing public API authentication, knowledge-base scope checks, and `MAINTENANCE` permission remain mandatory for evaluation runs and profile mutation.
- Internal ingest and QA APIs continue to require the internal key when configured.
- Profile filters are derived from validated enums, never copied as arbitrary Milvus expressions from a client request.
- Stored gate decisions are recomputed by the API from persisted evaluation results; the Web client cannot submit or override gate status.

## Testing Strategy

### Worker

- Universal ingest produces the five entry groups with correct scopes and contiguous chunk indexes.
- QA failure still indexes original and child routes.
- Legacy dual-write occurs only while the knowledge base is not v2-ready.
- Milvus v2 schema creation, validation, insertion, deletion, and profile filters are covered.
- Weighted RRF is deterministic and Combined handles an empty route.
- Hybrid corpus filtering excludes entries outside the requested profile.

### API

- V2 collection schema and validated profile expressions are covered in `MilvusVectorServiceTest`.
- Vector profile filtering and Combined route fusion are covered in `RetrievalServiceTest`.
- Ingest completion, deletion, and reindex revision changes are covered.
- Readiness cutover and legacy cleanup compensation are covered.
- Gate metrics, minimum-data rules, stale revisions, regression decisions, and current passes are covered.
- Knowledge base creation and profile update enforcement are covered at service, controller, and authorization layers.

### Web

- Knowledge base settings render readiness and current/stale/blocked/passed gate states.
- Profile selection cannot present a blocked candidate as enabled.
- Evaluation comparison renders metrics, deltas, and reason text.
- Existing profile controls and RAG eval workflows remain covered.

### Final Verification

Run API, Worker, and Web focused suites sequentially with the existing Maven memory limits. Run the Web production build, `git diff --check`, inspect generated files, and confirm no task-started Java, Node, or Python processes remain.

## Rollout Rule

`CLASSIC` remains the only unconditional default. A non-classic profile becomes selectable as the knowledge base default only after its current-revision gate is `PASSED`. P2 completes enforcement and diagnostics; it does not authorize a global automatic profile change.
