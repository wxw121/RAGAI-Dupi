# V1.4.1 Upload Governance Design

## Goal

V1.4.1 makes document upload and ingest safe under retry, cancellation, queue crashes, stale worker callbacks, and tenant quota pressure. The release keeps V1.4.0 recovery guarantees intact while moving upload governance from best-effort request handling into persisted, auditable state.

## User-visible outcomes

- Users can see remaining upload quota before sending files.
- Each file uploads independently with bounded browser concurrency and a stable idempotency key.
- A failed or cancelled file does not poison the rest of the batch.
- Cancelling a queued ingest job becomes terminal immediately.
- Cancelling a running ingest job requests cancellation and the worker stops at safe checkpoints.
- Retried jobs use a new execution ID so late callbacks from earlier attempts are ignored.
- Failed or dead-letter ingest terminal states emit a separate notification event.
- User cancellation does not send failure notifications.

## Existing system shape

- API is Spring Boot with PostgreSQL/Flyway, Redis queue, MinIO object storage, and Milvus vector cleanup.
- Worker is Python/FastAPI, consumes Redis, downloads MinIO objects, parses/chunks/embeds/indexes, then posts callbacks.
- Web is React/Vite and currently posts batch uploads through one request.
- V1.4.0 already added recovery archives, restore jobs, sparse recovery, ingest outbox, tombstones, and local release scan discipline.

## Architecture

### 1. Persistent tenant upload quota

Add a PostgreSQL quota ledger with two kinds of accounting:

- Retained quota: durable bytes and document count currently owned by a tenant/user.
- Upload window quota: bytes accepted during a rolling time window.

The API reserves retained quota before MinIO upload and commits it only after the document record, object key, ingest job, and ingest outbox event are durable. If MinIO or database work fails, retained quota is released. Window bytes are not refunded once the API accepts the request body because that protects upstream bandwidth and local resources.

Quota uses explicit reservations:

- PENDING: retained capacity reserved but not yet committed.
- COMMITTED: document exists and retained usage counts.
- RELEASED: capacity released after failure, deletion, or cancellation before commit.

Each upload request may include an Idempotency-Key. For the same tenant, user, knowledge base, and key, the API returns the original document/job response instead of creating a duplicate document. A conflicting key with a different file fingerprint returns 409. The implementation also stores `attemptId` / `attemptExpiresAt` on pending reservations, refreshes that lease across MinIO/database boundaries, and reconciles stale attempts by committing only when document, job, and outbox are durable. Retrying a released idempotent reservation rechecks retained quota but does not double-charge rolling-window bytes.

Recovery restore jobs bypass rolling upload window quota because they do not represent a user upload. After successful restore, retained usage is reconciled against restored document bytes.

### 2. Reliable ingest execution

Redis is a transport layer, not the source of truth. PostgreSQL owns whether a job can run.

Add execution fields to ingest jobs:

- executionId: current attempt ID included in Redis messages and callbacks.
- callbackSequence: highest accepted worker sequence.
- claimedBy / leaseExpiresAt / startedAt / completedAt: claim and lease bookkeeping.
- cancelRequestedAt: running cancellation state.

Worker Redis consumption changes from destructive BRPOP to a ready-to-processing move. The Worker only removes the processing item after the PostgreSQL callback has acknowledged a terminal state. If the Worker crashes after moving a message, the Worker processing reaper scans a bounded oldest tail, asks the API execution-state endpoint for `terminal`, `leaseExpired`, and `requeueEligible`, and atomically moves eligible payloads back to ready. Long operations heartbeat their lease before state-sensitive work.

Claim rules:

- Queued jobs can be claimed only when status is PENDING and executionId matches the message.
- Claim moves job to PROCESSING and stage PARSING with a lease.
- Retry rotates executionId and resets callbackSequence to 0.
- Recovery can requeue PENDING/QUEUED jobs and stale PROCESSING jobs whose lease expired.

### 3. Cancellation

Add an ingest cancellation endpoint available to the owner/operator.

- PENDING or queued job: transition to CANCELLED, release retained quota if not committed, cancel outbox event, and do not enqueue.
- PROCESSING job: transition to CANCEL_REQUESTED and persist cancelRequestedAt.
- Worker checks cancellation before parse, chunk, embedding batch, dense delete/index, sparse delete/index, and before final completion callback.
- If cancellation occurs after indexing starts, Worker deletes only execution-owned dense and sparse vectors before sending CANCELLED.
- User cancellation is terminal but not a failure and does not create failure notification events.

### 4. Stale callback protection

Every callback carries executionId and monotonically increasing sequence.

The API accepts only callbacks for the current job executionId and sequence greater than the stored callbackSequence. It also enforces legal forward transitions:

- PENDING -> PROCESSING
- PROCESSING -> COMPLETED
- PROCESSING -> FAILED
- PROCESSING -> CANCELLED
- CANCEL_REQUESTED -> CANCELLED
- FAILED -> retry creates new executionId and PENDING
- DEAD_LETTER -> retry creates new executionId and PENDING

Terminal states are never resurrected by old callbacks. Duplicate or stale callbacks return 200 with ignored=true so Worker retries do not amplify failures. Worker vector writes are also fenced with a per-document Redis lock and execution-scoped deterministic chunk IDs; cleanup uses exact `delete_by_ids` for dense and sparse stores instead of doc-wide deletion.

### 5. Failure notification outbox

Create an independent notification outbox for terminal ingest failures. It is separate from ingest outbox because ingest outbox controls work dispatch and must not be overloaded with user notification semantics.

Rules:

- First transition into FAILED or DEAD_LETTER writes one event in the same transaction.
- Event key is jobId:executionId:status.
- Duplicate terminal callbacks do not duplicate events.
- User cancellation does not create events.
- Delivery retries use bounded exponential backoff and persist last error/status.
- Notification webhook payload includes tenant, knowledge base, document, job, executionId, status, stage, and sanitized/truncated error. Delivery is HTTPS-only by default, blocks local/metadata hosts unless explicitly allowed, and can include `X-Dupi-Webhook-Secret`.

### 6. Web upload governance

The Web UI switches from one batch request to per-file orchestration.

- UploadZone receives quota and queue guardrail data.
- Each file gets an Idempotency-Key generated from name, size, modified time, and a local upload batch ID.
- Uploads run with limited concurrency.
- AbortController only aborts in-flight HTTP upload; once a document/job exists, user cancel calls the ingest cancel API.
- 409 idempotency conflicts, 429 quota/window throttles, and 413 file-too-large responses render per-file messages.
- Retry-After is displayed when present.
- Polling is serialized with AbortController so stale responses cannot overwrite newer state.
- Failed notifications are deduped by jobId:executionId:status.

## API contract

- GET /api/v1/upload-quota
  - Returns retained byte/document usage, configured limits, rolling window usage, and next retry time if throttled.
- POST /api/v1/knowledge-bases/{kbId}/documents
  - Accepts optional Idempotency-Key header.
  - Returns DocumentResponse with currentJob.
  - Returns 409 for idempotency key conflict.
  - Returns 429 for quota/window exhaustion with Retry-After if applicable.
  - Returns 413 for file size above configured retained quota or multipart max.
- POST /api/v1/knowledge-bases/{kbId}/ingest-jobs/{jobId}/cancel
  - Returns current IngestJobResponse.
- POST /api/v1/internal/ingest/jobs/{jobId}/claim
  - Worker claims execution before doing work.
- POST /api/v1/internal/ingest/jobs/{jobId}/lease
  - Worker refreshes lease.
- POST /api/v1/internal/ingest/status
  - Accepts executionId and sequence.
  - Returns {status:"ok", ignored:false} or {status:"ignored", ignored:true, reason:"..."}.
- GET /api/v1/internal/ingest/jobs/{jobId}/executions/{executionId}/state
  - Returns status, executionCurrent, terminal, leaseExpired, and requeueEligible for Worker heartbeats and processing-list reaping.

## Test strategy

- API unit tests cover quota reservation, idempotency conflict, deletion release, cancel transitions, execution rotation on retry, stale callback ignore, terminal callback dedupe, and notification outbox creation.
- API controller tests cover upload quota endpoint, Idempotency-Key, 409/429/413 response mapping, cancel endpoint, and callback ignored response.
- Worker tests cover ready-to-processing move, processing ACK, failed callback retaining processing item until acknowledged, cancellation checkpoints, cleanup on cancellation after indexing, executionId/sequence callback payloads, and stale callback ignored handling.
- Web tests cover per-file upload sequencing, Idempotency-Key generation, abort vs ingest cancel distinction, quota display, Retry-After display, and serialized polling.
- Release checks run API verify with JaCoCo, Worker pytest, Web tests/build, Pester release scan, docker compose config, diff checks, and local publish scan.

## Self-review

- Placeholder scan is clean.
- Scope is intentionally V1.4.1 and does not include new V1.4.2 features.
- The plan is buildable in vertical slices: persistence, API state machine, Worker transport, Web experience, notification delivery, release.
- Design risk is concentrated around transactional quota + external MinIO writes; the reservation state machine makes failures auditable and reversible.
- UX risk is concentrated around the difference between HTTP abort and ingest cancel; the plan names that distinction explicitly.
