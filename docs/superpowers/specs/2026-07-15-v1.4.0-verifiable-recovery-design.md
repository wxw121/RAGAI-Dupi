# V1.4.0 Verifiable Recovery Design

## Goal

V1.4.0 adds application-level, verifiable backup and restore for one knowledge base. An operator can archive the knowledge-base configuration, original MinIO objects, PostgreSQL records, evaluation assets, retrieval configuration, and Milvus vectors, then restore them into a new knowledge base with item-level checksums and an auditable result.

This closes the gap left by the V1.2 JSON export. That export remains a lightweight metadata migration format; it is not a disaster-recovery artifact.

## Release Sequence

V1.4 is delivered as two independently testable increments:

1. V1.4.0: verifiable archive and restore.
2. V1.4.1: tenant/user upload quotas, cancellation, and terminal-failure notification.

Only V1.4.0 is in scope for this design and its implementation plan. The two increments may share operational UI conventions, permissions, audit infrastructure, and outbox delivery, but they do not share state machines or database tables.

## Scope

An archive captures:

- knowledge-base configuration and lifecycle metadata required to reconstruct behavior;
- original document bytes from MinIO plus document and canonical chunk records;
- RAG evaluation cases, quality policy, and immutable retrieval profiles needed by the restored knowledge base;
- dense and active sparse Milvus vector records, including scalar payload required for retrieval;
- a versioned manifest containing counts, byte sizes, SHA-256 checksums, embedding model and dimension, collection schema, metric, and index settings.

The restored knowledge base receives a new identifier and remains unavailable until verification succeeds. Source identifiers are recorded for traceability but are never reused as target primary keys.

## Non-Goals

- PostgreSQL point-in-time recovery or cross-region replication.
- Infrastructure-level volume snapshots or orchestration of external backup products.
- Restoring into, overwriting, or mutating an existing knowledge base.
- Archiving chat sessions, user accounts, roles, audit history, transient ingest jobs, or transient migration jobs.
- Automatic scheduled retention policies. V1.4.0 supports explicit archive deletion; scheduling can be added after production usage is measured.
- Combining multiple knowledge bases into one archive.
- Upload quotas and cancellation, which belong to V1.4.1.

## Selected Storage Model

Recovery data is stored in a dedicated private MinIO bucket configured separately from the document bucket. The default bucket name is `dupi-recovery`. API credentials must have access to both buckets; neither bucket is public.

Each archive uses this prefix:

```text
archives/{tenantId}/{archiveId}/
  manifest.json
  records/knowledge-base.json
  records/documents.ndjson
  records/chunks.ndjson
  records/evaluation-cases.ndjson
  records/quality-policy.json
  records/retrieval-profiles.ndjson
  objects/{documentId}/{encodedFileName}
  vectors/dense.ndjson
  vectors/sparse.ndjson
```

Items are written separately rather than as one ZIP or TAR. This permits bounded-memory streaming, item-level retries, and checksum verification without rewriting a large package. `manifest.json` is written last. Its presence alone is insufficient: its database state, schema version, item list, counts, and checksums must also validate before an archive is recoverable.

Operators may download a completed archive as a streaming ZIP through an authenticated endpoint. The ZIP is assembled from the sealed prefix without buffering the complete package in API memory. It is only a transport view; the object set and verified manifest remain the canonical stored representation.

## Domain Model

### Recovery Archive

`recovery_archives` records the archive ID, tenant ID, source knowledge-base ID, state, schema version, bucket and prefix, captured knowledge-base revision, item/count summaries, error code and redacted error message, creator, timestamps, and manifest checksum.

States are:

```text
PREPARING -> CAPTURING -> VERIFYING -> COMPLETED
      |           |            |
      +-----------+------------+-> FAILED
```

`FAILED` archives remain diagnosable. Retry resumes the same archive ID after validating already written items; it does not create another logical archive. An operator may explicitly delete a failed or completed archive. Deletion records an audit event and removes the prefix idempotently.

### Recovery Items

`recovery_archive_items` stores a stable item key, type, source identifier, object key, expected byte size, SHA-256, item state, attempt count, and last redacted error. The unique key `(archive_id, item_key)` makes capture retry-safe.

Item states are `PENDING`, `WRITTEN`, `VERIFIED`, and `FAILED`. A retry verifies an existing object before deciding whether to reuse or rewrite it.

### Restore Job

`recovery_restore_jobs` records the archive ID, tenant ID, one target knowledge-base ID, state, progress counts, error information, actor, and timestamps.

States are:

```text
VALIDATING -> RESTORING_OBJECTS -> RESTORING_RECORDS
           -> RESTORING_VECTORS -> VERIFYING -> COMPLETED
                    |                   |
                    +-------------------+-> FAILED
```

The target knowledge base is allocated once when validation succeeds. Retrying a failed job continues with the same target. The target is marked `RESTORING` and is excluded from normal list, retrieval, upload, reindex, migration, evaluation, and chat operations until the job reaches `COMPLETED`.

`recovery_restore_items` maps each archive item to its target identifier and state. `(restore_job_id, archive_item_id)` is unique. Object writes, record inserts, and vector inserts use deterministic target keys so a repeated step verifies or replaces the same item rather than duplicating it.

## Consistent Archive Capture

Archive creation acquires the existing knowledge-base maintenance boundary and records a source revision. It waits for active ingest, reindex, evaluation, and sparse migration work to reach a terminal state. While capture is active, new document mutations, retrieval-profile activation, quality-policy mutation, evaluation-case mutation, reindex, and migration transitions are rejected with a structured maintenance response.

If the knowledge base cannot become quiescent within the configured timeout, archive creation fails before `CAPTURING`. The implementation must never label a mixed active-ingest snapshot as recoverable.

After quiescence:

1. Snapshot PostgreSQL records under a repeatable-read transaction and write record items.
2. Stream every referenced original object to the recovery bucket while calculating SHA-256.
3. Page through dense and active sparse Milvus data and write vector items without loading the whole collection into memory.
4. Build the manifest from persisted item evidence.
5. Re-read and verify every item, then write and verify the final manifest.
6. Mark the archive `COMPLETED` and release the maintenance boundary.

Any failure marks the archive `FAILED`, preserves item evidence, and releases the maintenance boundary. Errors exposed through APIs and audit logs are redacted and never include credentials, signed URLs, raw vectors, or provider configuration.

## Restore and Verification

Validation checks the manifest schema version, manifest checksum, tenant ownership, all item checksums and sizes, collection schema, metric/index configuration, embedding model and dimension, and declared counts. Validation completes before a target knowledge base is made visible.

Restore then:

1. Creates the target knowledge base in `RESTORING` state with a new ID and a conflict-safe restored name.
2. Restores original objects using deterministic target object keys.
3. Inserts remapped document, chunk, evaluation, policy, and retrieval-profile records through recovery-specific services that enforce restore idempotency.
4. Creates or validates versioned dense and sparse Milvus schemas before inserting remapped vectors.
5. Verifies MinIO bytes, PostgreSQL counts and relational ownership, vector counts and payload checksums, embedding compatibility, and representative retrieval behavior.
6. Marks the target ready and the restore job `COMPLETED` only after every verification passes.

A failed restore remains hidden and retryable. Explicit abandonment deletes the hidden target through compensating, idempotent cleanup and records an audit event. No automatic cleanup runs on failure because evidence must remain available for diagnosis.

## API and Permissions

All routes are tenant-scoped below `/api/v1/knowledge-bases` or `/api/v1/ops/recovery` and require normal resource authorization plus a new `KB_RECOVERY` permission.

Minimum API surface:

- create, list, inspect, retry, download, and delete archives;
- create, list, inspect, retry, and abandon restore jobs;
- expose item progress and redacted verification failures;
- reject cross-tenant archive and restore access regardless of supplied identifiers.

Every create, retry, completion, validation failure, deletion, and abandonment emits an audit event. Archive download uses authenticated streaming and is audited; the API does not return permanent public or presigned URLs.

## Web Experience

The knowledge-base operations area gains a `Recovery` section that follows the existing compact tables, dialogs, badges, and toast patterns. It shows archive state, captured time, size, counts, manifest checksum, actor, and available actions. Restore is a confirmation flow that states a new knowledge base will be created.

Progress polling must not resize the layout. Failed items show a redacted reason and retry action. A restored knowledge base appears in normal navigation only after verification completes. Global operations pages may link to failed recovery jobs but do not duplicate the full workflow.

## Configuration and Operations

New configuration includes recovery bucket name, quiescence timeout, item page size, and maximum concurrent archive/restore jobs. Defaults must be conservative for the single-node Compose deployment. Startup ensures the private recovery bucket exists but does not make it public.

The runbook documents bucket access policy, storage sizing, archive download, restore rehearsal, failure diagnosis, explicit deletion, and the distinction between application recovery and infrastructure backups. Production deployment still requires independent PostgreSQL, MinIO, etcd, and Milvus infrastructure backups.

## Failure Handling

- Cross-system writes are not globally transactional. Stable item keys, persisted progress, deterministic target keys, and compensation make retries safe.
- A missing or altered archive object fails validation before restore writes begin.
- A source mutation race detected by revision comparison fails archive verification.
- A vector schema, model, dimension, count, or payload mismatch blocks readiness.
- Process restarts leave non-terminal jobs resumable by an explicit retry; no second job may operate on the same archive or target concurrently.
- Deletion and abandonment are idempotent and report partial cleanup as diagnosable failure rather than claiming success.

## Testing and Release Gates

Unit and service tests cover state transitions, tenant isolation, permission checks, stable item keys, checksum validation, redaction, retry, and compensation. Controller tests cover structured errors and authorization.

Integration coverage uses real PostgreSQL, MinIO, and Milvus to:

1. create a fixture knowledge base containing original objects, chunks, evaluation assets, dense vectors, and sparse vectors;
2. archive and restore it;
3. compare object SHA-256 values, relational counts and ownership, vector counts and payload checksums;
4. run the same retrieval cases against source and target;
5. inject a mid-capture and mid-restore failure, retry, and prove that no duplicate target, object, record, or vector is created;
6. corrupt one archive item and prove restore is blocked before the target becomes ready.

Browser E2E covers archive creation, progress, completed details, restore confirmation, restored knowledge-base visibility, and a validation failure. The release gate also runs API, Worker, Web, production build, Compose validation, secret scanning, and `git diff --check`.

V1.4.0 is not complete until a real recovery rehearsal artifact records durations, item counts, byte counts, checksums, retrieval comparison, and cleanup result for a non-trivial fixture knowledge base.

## V1.4.1 Boundary

The next increment introduces durable quota reservations with `PENDING`, `COMMITTED`, and `RELEASED` states; per-tenant and per-user retained-byte/document limits; accepted-upload window accounting; per-file idempotency keys; browser cancellation; stale-reservation reconciliation; and terminal ingest-failure notification through the existing outbox boundary.

V1.4.0 must not pre-create quota tables or mix quota accounting into recovery jobs. Restore writes are privileged recovery operations and do not consume upload-window quota; after a successful restore, retained document and byte usage will be reconciled into V1.4.1 accounting when that increment is introduced.
