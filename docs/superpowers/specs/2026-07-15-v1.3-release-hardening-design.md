# V1.3 Release Hardening Design

## Summary

V1.3 is functionally complete, but release evidence is too narrow: the retrieval benchmark has one case, reranking is materially slower than vector and hybrid retrieval, the Milvus upgrade has not been rehearsed against a production-sized topology, and the new Python/image supply chain has no recorded scan. This hardening package turns those gaps into repeatable release gates and adds a guarded Web surface for the existing Sparse Migration state machine.

## Chosen Approach

Extend the existing V1.3 boundaries instead of introducing another retrieval platform or a separate operations application. The benchmark remains driven by the current RAG evaluation API, the Worker owns model lifecycle and Hugging Face cache behavior, deployment scripts own backup and upgrade rehearsal, and the knowledge-base detail page hosts the migration controls.

Two alternatives were rejected:

1. Documentation-only sign-off would record manual procedures but would not make benchmark coverage, backups, or scans reproducible.
2. A larger platform change, such as replacing the reranker stack or adding a dedicated operations service, would expand production risk during release hardening.

## Real-Corpus Benchmark

The repository will contain a deterministic 30-case seed manifest that can be expanded to 100 cases without changing the runner. It will cover six balanced categories with five cases each:

- Chinese direct and paraphrased queries.
- English direct and paraphrased queries.
- Exact identifiers, version strings, and error-code keywords.
- Semantic rewrites that avoid source wording.
- No-answer queries that must return no asserted evidence.
- Conflicting documents where the expected source and disambiguating keyword identify the authoritative result.

Every case has a stable `caseKey`, query, category, expected result contract, and source fixture reference. No-answer cases pass only when no hit crosses the configured evidence assertion; they must not be converted into ordinary keyword cases. Conflict cases require both the expected file and a disambiguating token.

The benchmark imports or reconciles the manifest idempotently, then runs VECTOR, HYBRID, and HYBRID+RERANK against the same case set. The JSON report records corpus and case counts, category coverage, profile snapshots, pass and assertion rates, P50/P95 latency, fallback rate, and per-case rank changes. Release validation fails when fewer than 30 enabled cases exist, a required category is empty, a run is blocked, sparse retrieval falls back, or required stage-rank evidence is absent.

## Rerank Performance

The Worker will make reranker lifecycle explicit:

- `RERANK_MODEL` selects the model and defaults to the existing model for compatibility.
- `RERANK_WARMUP_ENABLED` loads the model during Worker startup and performs one small inference before readiness succeeds.
- `HF_HOME`, `SENTENCE_TRANSFORMERS_HOME`, and the model cache share a persistent Docker volume so restarts do not download weights again.
- Readiness reports reranker availability without exposing local paths or credentials.
- Loading remains process-global and thread-safe; a failed warmup makes rerank unavailable while vector/hybrid retrieval remains healthy and observable.

The benchmark will measure at least one cold run and repeated warm runs. The release report compares the current model with one smaller candidate using identical cases and candidate limits. The smaller model is adopted only if it meets the configured quality gate and the agreed latency target; otherwise the current model remains and its measured latency is documented. The default release target is warm P95 at or below 250 ms for HYBRID+RERANK on the rehearsal environment, with no retrieval-quality regression versus the current reranker.

## Production Upgrade Rehearsal

A PowerShell orchestration script will operate on an explicitly named Compose project and output directory. It refuses to run without required credentials, sufficient disk space, healthy dependencies, and a destination outside live data directories.

Before Milvus 2.4.1 is stopped, the rehearsal captures:

- PostgreSQL logical dump plus restore metadata.
- etcd snapshot.
- MinIO object mirror and inventory.
- Milvus data volume archive and collection/schema inventory.
- SHA-256 checksums and a manifest containing image versions, timestamps, counts, and command outcomes.

The rehearsal restores those artifacts into a clean, production-equivalent topology, upgrades Milvus to 2.5.4 with a compatible PyMilvus client, runs health and count checks, executes sparse backfill, shadow validation, Cutover, and the 30-case benchmark, then exercises Rollback to the prior passing profile. Success requires matching PostgreSQL/object/chunk/vector counts, valid checksums, 100% migration coverage, passing quality gates, and successful retrieval before and after rollback. The script never targets production by default and requires an explicit acknowledgement flag for any non-rehearsal environment.

## Dependency and Image Scanning

The repository will provide a repeatable release scan that produces machine-readable and human-readable artifacts:

- Python dependency vulnerabilities, including PyTorch and sentence-transformers.
- License inventory with an explicit deny list for incompatible licenses.
- Worker image vulnerability scan by severity.
- Worker image size and layer summary.

The preferred local tools are `pip-audit` for Python packages, `syft` for SBOM/license inventory, and `trivy` for image vulnerabilities and size context. Missing tools fail the release scan with installation guidance; they do not silently skip checks. The gate fails on unfixed CRITICAL vulnerabilities, denied licenses, or a Worker image larger than the documented budget. HIGH findings may be accepted only through a dated exception recorded in the scan report.

## Sparse Migration Operations Page

The existing knowledge-base detail page gains a `Sparse Migration` operations panel beside retrieval profiles. It lists migration ID, profile, state, indexed/source coverage, legacy fallback state, last update, and failure reason. A compact state timeline makes the current phase and next valid action scannable.

Only actions valid for the current state are enabled. Start requires selecting an immutable retrieval profile. Backfill and shadow validation show progress and refresh status. Cutover is disabled until the API reports all acceptance checks satisfied, and requires a confirmation dialog summarizing coverage, gate status, latency ratio, fallback delta, and rollback profile. Legacy fallback is a labeled switch available only during dual-write and shadow validation. Failed migrations expose retryable actions and the server error without offering invalid state transitions.

The UI calls the existing migration API through typed client functions and treats the server as the state-transition authority. It does not synthesize migration state or bypass API validation. API errors appear in the existing toast/error pattern, destructive or production-impacting commands require confirmation, and controls remain usable on narrow screens without hiding status evidence.

## Testing and Evidence

- Worker unit tests cover singleton model loading, warmup success/failure, readiness, cache configuration, and rerank output.
- API and script tests cover manifest validation, no-answer/conflict semantics, backup preflight, manifest checksums, rehearsal refusal rules, and scan exit codes where practical.
- Web component tests cover state-specific actions, confirmation details, refresh, errors, and disabled Cutover.
- Browser E2E covers starting a migration, backfill, shadow validation, guarded Cutover, completion, fallback toggling, and a failed migration.
- Final verification runs API tests, Worker tests, Web tests/build, browser gate, 30-case benchmark, release scan, `git diff --check`, and documentation consistency checks.

## Documentation and Delivery

README, deployment documentation, E2E documentation, progress, and todo records will be updated from the final diff and generated evidence. The operations runbook will include cache persistence, warmup observability, backup/restore commands, upgrade acceptance criteria, rollback, scan policy, and artifact locations.

Implementation is delivered as three reviewable commits after a final diff review:

1. `feat(rag): harden sparse migration and rerank operations` for API, Worker, deployment, and migration console behavior.
2. `test(rag): expand retrieval benchmark and release gates` for the 30-case corpus, benchmark, rehearsal/scan automation, and test coverage.
3. `docs(v1.3): document release hardening and operations` for synchronized user and operator documentation plus non-generated evidence summaries.

Generated reports containing environment-specific IDs remain untracked unless they are sanitized fixtures. Branches use the `dev/` prefix; the active delivery branch is `dev/v1.3-rag-quality-recovery`.

## Open Risks

- A 30-case repository corpus improves representativeness but does not replace production query sampling. Expansion toward 100 cases remains an ongoing quality task.
- The 250 ms rerank target depends on rehearsal hardware. A documented exception may retain the current model only when quality is more important and the business latency budget explicitly permits it.
- A local Compose rehearsal validates the procedure, not infrastructure-specific production backup services. Production credentials, retention, encryption, and restore ownership remain deployment responsibilities.
- CVE databases and model artifacts require network access during the scan/build phase; release evidence must record database timestamps and immutable image digests.
