# V1.2.1 E2E Isolation and Cleanup Design

> **中文说明：** 本设计把真实浏览器回归数据隔离到 `e2e` 租户，并在成功运行后删除测试资源；失败时仅保留诊断所需的数据和标识符。

## Goal

Keep repeated real-browser regression runs out of the `default` tenant and remove all successful-run test data. When a run fails, retain only the associated `e2e` tenant data and identifiers needed to diagnose it.

## Scope

- Add a narrowly restricted API for deleting an `e2e_*` test account.
- Run browser-gate business flows inside an ephemeral `e2e` tenant administrator session.
- Remove successful-run knowledge bases and temporary accounts automatically.
- Preserve failed-run evidence in the `e2e` tenant and expose its identifiers in Playwright output.

The release does not add a general account-delete control to the Web account-management page, does not delete audit records, and does not remove prior historical test data automatically.

## Account Deletion Boundary

Add `DELETE /api/v1/ops/accounts/{username}`.

The existing `/api/v1/ops/**` filter requires `OPS_ADMIN`. `AccountService` adds a second, non-bypassable policy check before deletion:

1. The username must start with the exact lowercase prefix `e2e_`.
2. The account tenant must be exactly `e2e`.

All other requests return a validation error, including `e2e_*` names in `default` or any ordinary account. On success, the account is physically removed from `user_accounts` and the service records `ACCOUNT_DELETE_E2E` against target type `ACCOUNT`. Audit logs remain, preserving the deletion event without leaving a login-capable account.

The API is intentionally not added to the general Web account UI. It is a test-infrastructure control and is safe to invoke only from automation or an administrator-operated recovery command.

## Browser Gate Lifecycle

Every run creates a single `runId` and derives these identifiers:

- administrator: `e2e_gate_<runId>` in tenant `e2e`, role `ADMIN`
- verification account: `e2e_account_<runId>` in tenant `e2e`
- knowledge base: `e2e-browser-<runId>` in tenant `e2e`

The existing configured administrator logs in first only to create `e2e_gate_<runId>`. It creates no knowledge base and no test document in `default`. The test then logs in as `e2e_gate_<runId>` and performs the existing hybrid-knowledge-base, document/chat/RAG-eval, audit, account, and role flows.

The test records the three identifiers in Playwright output. A completion flag is set only after every product assertion and browser-error gate passes. In the `finally` block:

- successful run: delete the knowledge base, delete `e2e_account_<runId>`, then delete `e2e_gate_<runId>` through the restricted API; assert the cleaned resources no longer appear.
- failed run: skip cleanup and attach the identifiers and current URL to test output so that the evidence remains inspectable under tenant `e2e`.
- cleanup failure: fail the run and attach the resources that remain. It must not silently report a successful gate.

Cleanup HTTP calls use the authenticated browser session and the same CSRF mechanism as the Web console. This keeps the test subject to the real authorization and CSRF boundary while avoiding extra UI features solely for test teardown.

## Testing

Backend tests cover successful deletion and rejection of non-`e2e_*` usernames or non-`e2e` tenants, plus `ACCOUNT_DELETE_E2E` audit recording. Controller tests cover the endpoint delegation and the existing `OPS_ADMIN` filter remains the outer permission gate.

Web API tests cover the DELETE client helper. The real Chromium gate proves the complete success path: it must create the E2E tenant resources, perform the existing browser checks, delete all three resources, and verify that the configured administrator's `default` knowledge-base list did not gain the run's knowledge base. An intentionally failed run is not part of the normal gate; the failure branch is covered with unit-level test-harness assertions and Playwright's retained trace/screenshot/video configuration.

## Operational Notes

Runs that failed before this release left `e2e-browser-*` data in `default`; they are outside automated cleanup scope. Administrators can inspect and remove those separately. After V1.2.1, normal gate success leaves only auditable `e2e` tenant audit records and no test knowledge bases or user accounts.
