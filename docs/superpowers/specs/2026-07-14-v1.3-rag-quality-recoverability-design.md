# V1.3 RAG Quality and Recoverability Design / RAG 质量与可恢复性设计

## Goal / 目标

V1.3 makes retrieval quality measurable and enforceable before changing production retrieval behavior. It then makes hybrid retrieval observable and production-ready, followed by upload governance and a verifiable recovery workflow.

V1.3 在改变生产检索行为前，将检索质量变为可度量、可比较、可阻断的信号；随后将混合检索升级为可观测、可用于生产的能力，并完成上传治理和可校验的恢复流程。

## Scope and Delivery Order / 范围与交付顺序

1. **P0-1: RAG quality gate / RAG 质量门禁**
2. **P0-2: Production hybrid retrieval tuning / 混合检索生产调优**
3. **P1-1: Upload governance / 上传治理**
4. **P1-2: Verifiable disaster recovery / 可校验完整灾备恢复**

Disaster recovery remains P1 for this release. It must be promoted to P0 before a production launch that depends on application-managed recovery.

本版本灾备恢复保持 P1。若后续生产上线依赖应用侧恢复能力，必须在上线前提升为 P0。

## Current Constraints / 当前约束

- RAG evaluation persists cases, runs, and per-case results, but stores no promoted baseline, trend metrics, or blocking decision.
- Hybrid retrieval currently calls the Worker, which fetches the complete knowledge-base corpus per request and constructs an in-memory `BM25Okapi` index before RRF and optional reranking. This is not a production sparse index.
- Existing export/import restores only knowledge-base configuration and evaluation cases. It deliberately does not restore original MinIO objects, document records, chunks, or vectors.
- Upload controls expose rate and queue guardrails, but have no durable tenant/user quotas, client cancellation, upload-failure event, or advanced operations filters.

- 现有 RAG 评估已持久化用例、运行和逐用例结果，但没有已晋升基线、趋势指标或阻断结论。
- 现有混合检索由 Worker 在每次请求中获取整库语料、内存构建 `BM25Okapi`，再做 RRF 和可选重排；这不是生产级 sparse 索引。
- 现有导出/导入只恢复知识库配置和评估用例，刻意不恢复 MinIO 原文件、文档记录、分块或向量。
- 上传控制目前只展示限流和队列护栏，没有持久化租户/用户配额、客户端取消、上传失败事件或高级运维筛选。

## P0-1: Quality Gate / 质量门禁

### Model / 数据模型

Each knowledge base has one mutable quality policy and zero or one active baseline. A baseline points to an immutable completed evaluation run; it is promoted explicitly by an authorized administrator and never changes automatically.

每个知识库拥有一份可修改的质量策略和零或一个当前基线。基线指向一次不可变的已完成评估运行，只能由具备权限的管理员显式晋升，绝不自动替换。

The policy contains:

- `minimumPassRate`: absolute pass-rate floor.
- `maximumPassRateDrop`: maximum allowed drop relative to the baseline.
- `maximumNewFailures`: number of previously passing baseline cases allowed to fail.
- `blockWhenUnbaselined`: whether gate execution without a baseline blocks.

策略包含：

- `minimumPassRate`：绝对通过率下限。
- `maximumPassRateDrop`：相对基线允许的最大通过率下降。
- `maximumNewFailures`：基线中通过、当前失败的用例数上限。
- `blockWhenUnbaselined`：无基线运行是否阻断。

Each completed run records immutable summary metrics: pass rate, eligible expected-file hit rate, eligible keyword hit rate, average hit count, fallback count, and latency percentile summary. Each result receives a comparison state: `IMPROVED`, `UNCHANGED`, `REGRESSED`, or `NEW`.

每次完成运行记录不可变汇总指标：通过率、适用用例的期望文件命中率、适用用例的关键词命中率、平均命中数、降级次数和延迟分位摘要。每条结果具备 `IMPROVED`、`UNCHANGED`、`REGRESSED` 或 `NEW` 对比状态。

### Gate Decision / 门禁判定

The service returns `PASS`, `WARN`, `BLOCKED`, or `UNBASELINED`. A run is `BLOCKED` when it violates an absolute threshold, exceeds baseline pass-rate drop, or exceeds new-failure allowance. `WARN` is diagnostic-only and does not produce a non-zero process exit code.

服务返回 `PASS`、`WARN`、`BLOCKED` 或 `UNBASELINED`。违反绝对阈值、超过基线通过率降幅或新增失败数上限时为 `BLOCKED`。`WARN` 仅用于诊断，不产生非零退出码。

`scripts/rag-quality-gate.ps1` runs an evaluation and exits non-zero only for `BLOCKED` and policy-configured `UNBASELINED`. It is the CI/release integration point; normal UI runs still return the complete diagnostic result.

`scripts/rag-quality-gate.ps1` 运行评估，并仅在 `BLOCKED` 或策略规定需阻断的 `UNBASELINED` 时返回非零退出码。该脚本是 CI/发布接入点；普通 UI 运行仍返回完整诊断结果。

### Deterministic Comparison Rules / 确定性对比规则

`caseKey` is the stable comparison identity. A run also stores a fingerprint of the normalized query and every assertion field. A current case with a missing baseline key or a changed fingerprint is `NEW`; baseline-only removed cases are reported separately and never count as a new failure. Only matching key-and-fingerprint pairs contribute to regression metrics.

`caseKey` 是稳定的对比标识。每次运行还保存规范化问题和全部断言字段的指纹。当前用例缺少基线键或指纹变化时为 `NEW`；只存在于基线的已删除用例单独报告，绝不计入新增失败。只有键和指纹都相同的用例对参与退化指标计算。

`BLOCKED` means an absolute policy threshold, permitted pass-rate drop, or permitted new-failure count was violated; it also applies to an unbaselined run when `blockWhenUnbaselined` is true. `UNBASELINED` means all absolute thresholds pass but no active baseline exists and the policy is non-blocking. `WARN` means no blocking rule failed but the current and baseline case sets are not fully comparable. `PASS` requires an active baseline, full comparability, and every blocking rule to pass. A baseline may be promoted only from a completed `PASS` run, or from a completed `UNBASELINED` run with all absolute thresholds passing to establish the first baseline; `WARN` and `BLOCKED` runs are never promotable.

`BLOCKED` 表示违反绝对策略阈值、允许通过率降幅或允许新增失败数；当 `blockWhenUnbaselined` 为真时，无基线运行也为 `BLOCKED`。`UNBASELINED` 表示绝对阈值均通过但没有活动基线，且策略不阻断。`WARN` 表示没有阻断规则失败，但当前与基线用例集不可完全比较。`PASS` 要求存在活动基线、完全可比且所有阻断规则通过。只有已完成的 `PASS` 运行，或用于建立首个基线且全部绝对阈值通过的已完成 `UNBASELINED` 运行可以被晋升；`WARN` 与 `BLOCKED` 运行永不可晋升。

### UI / 界面

The RAG evaluation tab shows policy controls, active baseline, a compact metric trend, gate decision, and a per-case baseline comparison. Promoting a baseline requires an explicit confirmation and is available only after a completed passing run.

RAG 评估页展示策略控件、当前基线、紧凑的指标趋势、门禁结论及逐用例基线对比。仅已完成且通过的运行可被晋升为基线，操作需显式确认。

## P0-2: Production Hybrid Retrieval / 混合检索生产调优

### Alternatives and Decision / 方案比较与决策

1. Tune the current per-request in-memory BM25 implementation. It is low effort but retains full-corpus fetch and cannot satisfy the sparse-index production requirement.
2. **Chosen: migrate sparse retrieval to a Milvus sparse/BM25 inverted field**, with a temporary, observable Worker fallback during migration. It removes per-request corpus construction while retaining the current Milvus deployment boundary.
3. Add OpenSearch/Elasticsearch. It is more flexible but adds an unnecessary production dependency for this release.

1. 调优当前每请求内存 BM25，实现成本低，但保留整库拉取，无法满足生产 sparse 索引要求。
2. **选定：迁移到 Milvus sparse/BM25 倒排字段**，并在迁移期间保留可观测的 Worker 回退。该方案移除每请求语料构建，同时维持当前 Milvus 部署边界。
3. 引入 OpenSearch/Elasticsearch，灵活性更高，但为本版本引入了不必要的生产依赖。

### Retrieval Profile / 检索配置档案

Retrieval configuration is versioned as a named profile. It captures vector candidate count, sparse candidate count, RRF constant, sparse index/search parameters, rerank enablement, rerank candidate limit, and final top-K. Runs persist the full profile snapshot, not only a boolean rerank flag.

检索配置以命名、可版本化的档案保存，包含向量候选数、sparse 候选数、RRF 常量、sparse 索引/搜索参数、是否启用 Rerank、Rerank 候选上限和最终 top-K。评估运行保存完整档案快照，而不只保存 Rerank 布尔值。

Each knowledge base has exactly one active production profile. A profile can be activated only when a completed quality-gate run uses its exact immutable snapshot and returns `PASS`. Activation is audited and immediately makes the API pass that profile to the Worker. A rollback is an audited activation of a previously passing profile; it never mutates historical runs.

每个知识库恰有一份活动生产配置档案。只有当已完成的质量门禁运行使用该档案的精确不可变快照且返回 `PASS` 时，档案才能启用。启用操作会被审计，并立即让 API 将该档案传递给 Worker。回滚是对先前通过档案的一次审计启用，绝不修改历史运行。

The retrieval response adds stage explanations: vector candidate rank, sparse candidate rank, fusion score/rank, rerank score/rank, applied profile version, and any fallback reason. Raw vectors and provider credentials are never exposed.

检索响应增加阶段解释：向量候选排名、sparse 候选排名、融合分数/排名、重排分数/排名、应用的配置档案版本和任何回退原因。不得暴露原始向量或供应商凭据。

### Benchmark / 压测与对比

The benchmark script runs the same baseline dataset through `VECTOR`, `HYBRID`, and `HYBRID+RERANK` profiles. It reports quality-gate metrics, P50/P95 latency, fallback rate, corpus size, profile snapshot, and per-case rank changes. The UI compares two runs or profiles using these immutable snapshots.

压测脚本以同一基线数据集运行 `VECTOR`、`HYBRID` 和 `HYBRID+RERANK` 配置，报告质量门禁指标、P50/P95 延迟、回退比例、语料规模、配置快照和逐用例排名变化。UI 基于这些不可变快照比较任意两次运行或两个配置档案。

### Sparse Migration / Sparse 迁移

Migration uses a new versioned Milvus collection/schema rather than altering a live collection in place. It has explicit states: `PREPARING`, `BACKFILLING`, `DUAL_WRITING`, `SHADOW_VALIDATING`, `CUTOVER`, `COMPLETED`, and `FAILED`. Backfill reads canonical PostgreSQL chunks, validates embedding dimension and sparse schema/index parameters, and writes the new collection idempotently. New and reindexed documents dual-write until cutover.

迁移使用新的版本化 Milvus collection/schema，而不就地修改在线集合。状态明确为 `PREPARING`、`BACKFILLING`、`DUAL_WRITING`、`SHADOW_VALIDATING`、`CUTOVER`、`COMPLETED` 和 `FAILED`。回填读取 PostgreSQL 中的规范分块，校验 embedding 维度及 sparse schema/索引参数，并幂等写入新集合。新上传和重建索引的文档在切换前双写。

Cutover requires 100% expected chunk coverage, matching dimensions, no blocking quality-gate result for the candidate profile, P95 latency no worse than 1.25 times the current production baseline, and no higher fallback rate. The legacy Worker `BM25Okapi` route remains behind an audited feature flag only during dual-write and shadow validation; it is disabled after one stable release with these acceptance criteria met, but retained code is not deleted until a later release.

切换要求预期分块覆盖率 100%、维度一致、候选档案没有被质量门禁阻断、P95 延迟不高于当前生产基线的 1.25 倍，且回退率不升高。旧 Worker `BM25Okapi` 路径仅在双写和影子验证期间通过可审计特性开关保留；满足这些验收条件并稳定一个版本后禁用，但代码留待后续版本删除。

## P1-1: Upload Governance / 上传治理

### Quotas and Cancellation / 配额与取消

Durable quota accounting enforces per-tenant and per-user limits for retained bytes, retained document count, and upload bytes in a time window. Upload admission atomically reserves capacity before object storage; failures and deletions release reservations. Existing rate limiting remains a separate short-window protection.

持久化配额核算对租户和用户分别执行已保留字节数、已保留文档数和时间窗口上传字节数限制。上传准入在写入对象存储前原子预占容量；失败和删除释放预占。现有限流继续作为独立的短窗口保护。

The Web batch uploader uses `AbortController`. Cancelling stops in-flight HTTP requests and prevents unstarted files in the batch; documents already accepted remain auditable and proceed through normal ingest handling.

Web 批量上传使用 `AbortController`。取消会中止正在传输的 HTTP 请求并阻止批次中未开始的文件；已被服务端接收的文档保留可审计状态，并继续走正常摄入处理。

Every file carries an idempotency key. Storage/document reservations have `PENDING`, `COMMITTED`, and `RELEASED` states. Retained-byte and retained-document capacity is committed only after the object and document record are accepted, then released on terminal storage failure or deletion. Window upload bytes measure accepted upload attempts and are not released after server acceptance, preventing cancellation/retry quota bypass; retries with the same idempotency key do not charge twice. A scheduled reconciler releases stale `PENDING` reservations after verifying that neither object nor document was accepted.

每个文件携带幂等键。存储/文档预占具有 `PENDING`、`COMMITTED` 和 `RELEASED` 状态。已保留字节和已保留文档容量只在对象与文档记录被接收后提交，并在最终存储失败或删除时释放。时间窗口上传字节按服务端已接收的上传尝试计量，接收后不因取消/重试释放，以防绕过配额；相同幂等键的重试不重复计费。定时协调器在确认对象和文档均未被接收后释放过期 `PENDING` 预占。

### Notifications and Operations / 通知与运维

Upload and ingest terminal failures create an outbox notification event and use the existing Webhook delivery boundary. Operations filtering adds tenant, actor, knowledge base, document, status, stage, and time range without exposing cross-tenant data.

上传和摄入的最终失败创建 outbox 通知事件，并复用现有 Webhook 投递边界。运维筛选增加租户、操作者、知识库、文档、状态、阶段和时间范围，同时不得暴露跨租户数据。

## P1-2: Verifiable Disaster Recovery / 可校验完整灾备恢复

### Archive / 归档

A versioned recovery archive contains a manifest, original MinIO objects, knowledge-base configuration, document and chunk records, evaluation assets, vector records, and SHA-256 checksums. The archive is distinct from the V1.2 metadata export and carries an explicit recovery schema version.

版本化恢复归档包含清单、MinIO 原始对象、知识库配置、文档和分块记录、评估资产、向量记录及 SHA-256 校验值。该归档不同于 V1.2 元数据导出，并带有明确的恢复 schema 版本。

### Restore State Machine / 恢复状态机

Restore always creates a new knowledge base and progresses through `VALIDATING`, `RESTORING_OBJECTS`, `RESTORING_RECORDS`, `RESTORING_VECTORS`, `VERIFYING`, `COMPLETED`, or `FAILED`. It validates manifest checksums, object bytes, document/chunk counts, vector count, vector dimension, and vector payload checksum. A failed restore remains diagnosable and retryable but is never exposed as ready.

恢复始终创建新知识库，并经过 `VALIDATING`、`RESTORING_OBJECTS`、`RESTORING_RECORDS`、`RESTORING_VECTORS`、`VERIFYING`、`COMPLETED` 或 `FAILED` 状态。流程校验清单校验和、对象字节、文档/分块数量、向量数量、向量维度和向量载荷校验和。失败恢复保持可诊断、可重试，但绝不作为可用知识库暴露。

Archive creation acquires a knowledge-base maintenance lock, waits for active ingest/reindex work to finish, and rejects new mutations until the manifest is sealed. The manifest records the captured revision, source collection schema, metric/index settings, embedding model and dimension, and per-object/vector checksums. An archive is rejected when quiescence cannot be reached; it never presents a mixed active-ingest snapshot as recoverable.

创建归档时会获取知识库维护锁，等待活动摄入/重建任务完成，并在清单封存前拒绝新的变更。清单记录捕获版本、源 collection schema、度量/索引设置、embedding 模型和维度，以及每个对象/向量的校验和。无法达到静止状态时归档直接失败；绝不将混合活动摄入快照标记为可恢复。

A restore job creates one target knowledge base once. Retrying the job resumes the same target and archive item manifest after compensating incomplete items; it never creates another target. Vector restore provisions or validates an exactly compatible versioned Milvus schema before inserting archived vector payloads. Schema, metric, index, model, dimension, count, and payload-checksum mismatches block restore before the target can become ready.

恢复任务只创建一次目标知识库。任务重试会在补偿未完成项后，基于同一目标和归档项清单继续执行，绝不创建另一个目标。向量恢复会先创建或校验完全兼容的版本化 Milvus schema，再插入归档向量载荷。schema、度量、索引、模型、维度、数量或载荷校验和不匹配时，恢复被阻断，目标不得成为可用状态。

Recovery operations use a dedicated permission and audit every archive, validation failure, retry, and completed restore. Cross-system writes cannot be globally transactional; compensation records and idempotent archive item keys make retries safe.

恢复操作使用独立权限，并审计每次归档、校验失败、重试和完成恢复。跨系统写入无法形成全局事务；补偿记录和幂等归档项键确保重试安全。

## Non-Goals / 非目标

- No LLM answer-quality judging in V1.3; the gate evaluates retrieval evidence only.
- No automatic promotion of a quality baseline.
- No OpenSearch/Elasticsearch dependency.
- No cross-region storage replication or point-in-time PostgreSQL restore orchestration.
- No silent partial recovery or direct reuse of source knowledge-base identifiers.

- V1.3 不做 LLM 回答质量裁判；门禁只评估检索证据。
- 不自动晋升质量基线。
- 不引入 OpenSearch/Elasticsearch。
- 不做跨区域存储复制或 PostgreSQL 时间点恢复编排。
- 不允许静默部分恢复，也不直接复用源知识库标识。

## Verification / 验证

- Service and controller tests cover policy calculation, baseline promotion authorization, blocked script exit code, profile snapshots, and explanation redaction.
- A real corpus benchmark proves comparable reports for the three retrieval modes and confirms sparse retrieval avoids request-time full-corpus fetch.
- Browser E2E covers baseline promotion, a blocked regression, quota rejection, upload cancellation, and operations filters with isolated E2E data.
- A recovery integration test exports a fixture knowledge base, restores it, and verifies object checksums, PostgreSQL counts, vector payload checksums, and retrieval behavior.

- 服务和控制器测试覆盖策略计算、基线晋升权限、阻断脚本退出码、配置档案快照和检索解释脱敏。
- 真实语料压测证明三种检索模式可生成可比报告，并确认 sparse 检索不再在请求时拉取整库语料。
- 浏览器 E2E 覆盖基线晋升、被阻断的退化、配额拒绝、上传取消和运维筛选，且继续使用隔离的 E2E 测试数据。
- 恢复集成测试导出固定知识库、恢复后校验对象校验和、PostgreSQL 计数、向量载荷校验和及检索行为。
