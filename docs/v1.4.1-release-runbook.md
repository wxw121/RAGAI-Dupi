# V1.4.1 Upload Governance Release Runbook

## Scope

V1.4.1 adds durable upload quota reservations, per-file idempotency, reliable ingest execution/ACK behavior, cancellation, stale callback protection, terminal-failure webhook delivery, and governed Web uploads. V1.4.0 recovery archives remain unchanged.

## Preflight

1. Back up PostgreSQL before Flyway V17. V17 creates `upload_quota_reservations`, `upload_window_events`, and `ingest_failure_notifications`, adds `documents.quota_reservation_id`, and adds execution/lease columns to `ingest_jobs`.
2. Confirm Redis has capacity for both the ready queue and `INGEST_PROCESSING_QUEUE`.
3. Configure the quota variables documented in the README. Defaults are 1 GiB retained bytes, 1,000 retained documents, and 256 MiB per one-hour rolling window per tenant/user. Retained usage counts active `PENDING + COMMITTED` reservations; `RELEASED` rows do not consume quota.
4. If operations needs external failure alerts, configure `INGEST_FAILURE_NOTIFICATION_WEBHOOK_URL`; delivery is skipped when it is blank. Tune `INGEST_FAILURE_NOTIFICATION_TIMEOUT_SECONDS`, `INGEST_FAILURE_NOTIFICATION_MAX_ATTEMPTS`, and `INGEST_FAILURE_NOTIFICATION_DISPATCH_CRON` for the target endpoint.
5. Use the same non-empty `DUPI_INTERNAL_KEY` for API and Worker in shared environments.
6. Do not delete the processing list during rollout. It is recovery evidence for jobs moved out of the ready list but not terminally acknowledged.

## Rollout

1. Deploy PostgreSQL/Flyway and API first.
2. Verify `GET /actuator/health` and `GET /api/v1/upload-quota` with an account that has `DOCUMENT_UPLOAD`.
3. Deploy Worker with `INGEST_PROCESSING_QUEUE`, `INGEST_LEASE_SECONDS`, and a unique `WORKER_ID`.
4. Deploy Web after API and Worker are healthy.
5. Upload one small document with an `Idempotency-Key`; replay the same request and confirm the same document/current job is returned.
6. Reuse the key with different content and confirm 409. Exercise an oversized file for 413 and an exhausted window for 429 plus `Retry-After`.
7. Cancel one queued job and one running job. Confirm queued becomes `CANCELLED`; running moves through `CANCEL_REQUESTED` to `CANCELLED`, with no failure notification.
8. Retry a failed job and confirm `executionId` changes. Send an old-execution or duplicate-sequence callback and confirm HTTP 200 with `ignored=true`.
9. When `INGEST_FAILURE_NOTIFICATION_WEBHOOK_URL` is configured, force one FAILED ingest and confirm a single webhook delivery; repeated terminal callbacks must not create another event. A failing webhook should move the event through `FAILED` retries and eventually `EXHAUSTED`.

## Operational Checks

- Compare `retainedBytesUsed` and `retainedDocumentsUsed` with active `PENDING + COMMITTED` reservations and live documents; include old pending reservations in reconciliation before changing quota totals manually.
- Investigate old `PENDING` reservations and old processing-list messages before changing quota totals manually.
- A `CANCEL_REQUESTED` job should keep polling until the Worker returns `CANCELLED`.
- `ingest_failure_notifications.event_key` is unique. Duplicate callbacks must not create another row; webhook delivery is retried from `PENDING`/`FAILED` rows until `DELIVERED` or `EXHAUSTED`.
- Recovery restore should create `COMMITTED` reservations for restored documents; abandoning a restore should release linked reservations with reason `Recovery restore abandoned`.
- Web transport abort only stops the current HTTP upload. If a response already created a document/job, use the ingest cancel endpoint.

## Rollback

1. Roll back Web first; the batch API remains available for compatibility.
2. Stop Worker before rolling back API so no V1.4.1 claim/lease/callback contract is sent to an older API.
3. Prefer a forward fix for API/database issues. V17 is additive, but dropping its tables/columns destroys quota and execution evidence.
4. If Worker rollout fails, retain the processing list, restore the V1.4.1 Worker, and let PostgreSQL execution/lease state decide whether work may resume.

## Release Boundary

Local merge/tag is permitted only after all gates above pass on the final diff. Remote push, pull request, or remote tag requires explicit authorization because it publishes workspace code externally.
