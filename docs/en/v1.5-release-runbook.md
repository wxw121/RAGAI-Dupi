## Rollback

1. Switch a knowledge base's default back to `CLASSIC`; this is unconditional and does not require candidate gate evidence.
2. Do not delete the Profile V2 collection while investigating a candidate-quality or latency regression. It is the shared source for all four profiles and for reproducible evaluation.
3. If Worker rollout fails before Profile V2 cutover, stop Worker, retain the reliable processing queue, restore the release Worker, and resume from persisted document/index state.
4. Prefer a forward fix for API/database failures. V18-V20 are additive, but manually changing `profile_index_activated`, `index_revision`, or document schema versions can re-enable an invalid fallback path.
5. After Legacy cleanup, rolling back to code that only understands the Legacy collection requires a verified archive/restore or a complete reindex. Do not clear the persisted cutover state to force fallback.
6. Roll back Web first, then Worker, then API only after confirming the older components understand the current queue payload and database schema.
