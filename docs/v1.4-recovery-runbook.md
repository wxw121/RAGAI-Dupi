# V1.4 Recovery Runbook

> **中文说明：** 本手册说明 V1.4.0 知识库归档与恢复的存储、权限、操作流程和验证要求。英文内容为规范正文，本说明用于帮助中文读者快速确认操作边界。

## Scope

V1.4.0 archives and restores one knowledge base, including original objects, relational records, evaluation/retrieval configuration, and dense/sparse vector payloads. It does not provide PostgreSQL PITR, cross-region replication, or scheduled retention.

## Storage And Access

Use a dedicated private MinIO bucket, default `dupi-recovery`. The API credential needs read/write/list/delete access to document and recovery buckets; neither bucket may be public. Canonical objects live below `archives/{tenantId}/{archiveId}/`. Downloaded ZIP files are transport views, not canonical storage.

Size storage from retained source objects plus serialized records/vectors and concurrent-archive headroom. Delete archives explicitly after the required retention period; V1.4.0 has no retention scheduler.

## Operator Workflow

1. Open a knowledge base and select **Recovery**.
2. Create an archive. Mutations are blocked while active ingest, evaluation, and sparse migration work becomes quiescent.
3. Wait for `COMPLETED`; inspect item count, bytes, and manifest checksum. Retry failures. Only completed archives can be downloaded.
4. Restore a completed archive. The target remains hidden while `RESTORING`; retry resumes the same target.
5. Follow the target link only after `COMPLETED`. Abandon a failed restore that should be removed.

Never place object keys, vectors, signed URLs, credentials, or raw provider failures in tickets or audit descriptions. Recovery errors are redacted.

## Rehearsal

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -Command "Import-Module Pester; Invoke-Pester scripts/tests/rehearse-v1.4-recovery.Tests.ps1"
$corruptObject = {
  param($objectKey)
  docker run --rm --network deploy_default --entrypoint /bin/sh `
    -e "MC_ACCESS=$env:MINIO_ACCESS_KEY" -e "MC_SECRET=$env:MINIO_SECRET_KEY" -e "MC_OBJECT=$objectKey" minio/mc `
    -c 'mc alias set local http://minio:9000 "$MC_ACCESS" "$MC_SECRET" >/dev/null && printf corrupted-recovery-object | mc pipe "local/dupi-recovery/$MC_OBJECT" >/dev/null'
  return ($LASTEXITCODE -eq 0)
}.GetNewClosure()
& scripts/rehearse-v1.4-recovery.ps1 `
  -BaseUrl http://localhost:8080 `
  -ApiKey $env:DUPI_API_KEY `
  -FixtureFiles examples/sample-knowledge.md,examples/sample-knowledge.md `
  -CorruptObject $corruptObject
```

The release artifact is `artifacts/v1.4-recovery/rehearsal.json`. It must record at least two documents, archive counts/bytes/checksum, object/record/vector verification, retrieval comparison, corruption blocking, durations, and cleanup. Every evidence flag must be true or the script exits non-zero. Policy-only validation, a run without `CorruptObject`, and Playwright discovery are not release evidence.

The final 2026-07-16 Compose run captured 2 documents as 9 items / 10,946 bytes and passed object/vector checksum recomputation, restored record counts, retrieval equivalence, canonical-object corruption rejection, and scoped cleanup. The artifact records `corruptionBlocked=true`.

## Release Scan

Build and scan the final Worker image from the repository root:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/scan-release.ps1 `
  -Image dupi-rag-worker:v1.4 `
  -OutputPath artifacts/v1.4-release-scan `
  -HighExceptionPath deploy/release-exceptions/v1.4.0.json
```

The image must pass `pip check`, import the real Worker/PyMilvus stack, use CPU-only PyTorch, run as UID/GID `65534`, stay below the 3 GiB budget, produce pip-audit, CycloneDX/Syft SBOM, Trivy, and summary artifacts, and pass the application-dependency license denylist. PyMilvus 2.5.18 stays on the Milvus 2.5 client line while removing the deprecated `pkg_resources` dependency, which permits the patched packaging toolchain. The scan accepts `pip-audit` on `PATH` or falls back to `python -m pip_audit`. `deploy/release-exceptions/v1.4.0.json` must declare the normalized image release and contains exact package/CVE matches only for findings without an upstream fix; it expires on `2026-08-15`. New, fixable, inactive, unmatched, wrong-release, or expired findings block release. `-TrivySkipDbUpdate` is permitted only after separately refreshing the local database.

The final 2026-07-17 release scan passed with a 634,098,035-byte image, 3 CRITICAL and 19 HIGH Trivy findings plus 1 pip-audit finding, all 23 covered by the dated exception. `summary.md`, `worker-trivy.json`, and `trivy-version.json` retain the image digest and vulnerability-database timestamp without hard-coding a digest that changes when BuildKit regenerates provenance. Runtime verification reported `uid=65534`, `gid=65534`, `torch=2.10.0+cpu`, `pymilvus=2.5.18`, and `cuda=False`.

## Failure Handling

- Quiescence timeout: let active work finish, diagnose stuck jobs, retry the same archive.
- Capacity exceeded: reduce concurrent work; raise the configured limit only after measuring dependency and memory load.
- Archive verification failure: do not restore it. Retain evidence until diagnosis is complete.
- Restore verification failure: the target stays hidden. Correct the dependency and retry the same job, or abandon it.
- Partial abandon failure: do not report cleanup complete while object, record, or vector residue remains.

Flyway V15 and V16 make completed restore-job and restore-item evidence cascade with explicit archive deletion. Apply both migrations before relying on archive-retention cleanup.

## Infrastructure Backups

Application recovery is one layer. Production still requires independent, tested backups for PostgreSQL, both MinIO buckets, etcd, and Milvus, with off-host retention. Preserve encryption keys and credentials through secret management, never in rehearsal artifacts.
