# V1.5.0 RAG Quality Upgrade Release Runbook

Status: release. Create the `v1.5.0` tag only after every final gate in this document passes on the release commit and the merged `main` CI is green.

## Scope

V1.5.0 adds knowledge-base-level `CLASSIC`, `PARENT_CHILD`, `QA_ASSISTED`, and `COMBINED` retrieval profiles without replacing the existing `VECTOR`, `HYBRID`, and `RERANK` execution modes. Worker ingest builds one filterable Profile V2 superset containing original, parent, child, and QA entries. Non-classic profiles can become a knowledge-base default only after the current index revision passes a candidate-vs-classic quality gate.

## Compatibility

- Existing requests that do not select a retrieval profile continue to use `CLASSIC`.
- `VECTOR`, `HYBRID`, and optional reranking retain their existing API behavior.
- QA generation failure does not fail original-document ingestion.
- Profile selection changes the retrieval entry route; it does not rebuild an already-ready Profile V2 index.
- A restored knowledge base retains its profile collection data, gate state, and cutover state through the V1.4 recovery contract.

## Preflight

1. Back up PostgreSQL, MinIO, etcd, and Milvus before Flyway V18-V20. V18 adds retrieval index modes, V19 adds Profile V2 revision/readiness and evaluation evidence, and V20 persists Profile V2 cutover state.
2. Record each knowledge base's document count, completed-document count, current default profile, active retrieval profile, index revision, and current Milvus collection counts.
3. Configure the same non-empty `MILVUS_PROFILE_COLLECTION` for API and Worker. It must not reuse `MILVUS_COLLECTION` or another environment's collection.
4. Confirm Milvus has capacity for the original/parent/child/QA superset during the one-time migration and for temporary Legacy dual writes before cutover.
5. Keep `COMBINED_CHILD_WEIGHT=1.0` and `COMBINED_QA_WEIGHT=0.8` for the initial rollout unless a retained benchmark justifies different values.
6. Use the same non-empty `DUPI_INTERNAL_KEY` for API and Worker. Confirm the LLM endpoint used for QA generation is reachable, but do not block rollout solely because QA generation is unavailable.
7. Keep every existing knowledge base on `CLASSIC` until its Profile V2 index is ready and its current revision has a passing quality gate.

## Rollout

1. Deploy PostgreSQL/Flyway and API first. Verify `/actuator/health`, Flyway V18-V20, and the existing knowledge-base list before starting Worker.
2. Deploy Worker with `MILVUS_PROFILE_COLLECTION`, Combined weights, and the existing reliable queue settings. New ingestion dual-writes Legacy original vectors until the knowledge base reaches Profile V2 cutover.
3. Deploy Web after API and Worker are healthy. Verify the knowledge-base settings and RAG evaluation panels load before migrating data.
4. Select one non-critical knowledge base and run one reindex. Reindex rolls documents in place; do not delete the Profile V2 collection or clear all live chunks first.
5. Wait until every document is `COMPLETED` with `index_schema_version=2`. Confirm Profile V2 readiness, a non-empty index revision, and expected original/parent/child/QA counts before cutover cleanup.
6. Run at least three citation-evaluable cases for `CLASSIC` and each candidate profile. The candidate must not regress hit rate or citation pass rate, and the result must reference the current index revision.
7. Activate a non-classic default only when its decision is `PASSED`. Treat `BLOCKED`, `INSUFFICIENT_DATA`, `STALE`, `INDEX_NOT_READY`, and `NOT_EVALUATED` as hard stops; the API returns `409 retrieval_profile_gate_blocked` for blocked activation.
8. Expand rollout one knowledge base at a time. Start with `PARENT_CHILD`; enable `QA_ASSISTED` or `COMBINED` only where retained evaluation evidence shows a benefit.

## Rollback

1. Switch a knowledge base's default back to `CLASSIC`; this is unconditional and does not require candidate gate evidence.
2. Do not delete the Profile V2 collection while investigating a candidate-quality or latency regression. It is the shared source for all four profiles and for reproducible evaluation.
3. If Worker rollout fails before Profile V2 cutover, stop Worker, retain the reliable processing queue, restore the release Worker, and resume from persisted document/index state.
4. Prefer a forward fix for API/database failures. V18-V20 are additive, but manually changing `profile_index_activated`, `index_revision`, or document schema versions can re-enable an invalid fallback path.
5. After Legacy cleanup, rolling back to code that only understands the Legacy collection requires a verified archive/restore or a complete reindex. Do not clear the persisted cutover state to force fallback.
6. Roll back Web first, then Worker, then API only after confirming the older components understand the current queue payload and database schema.
