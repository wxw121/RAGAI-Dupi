# #范围

V1.4.0存档和恢复一个知识库，包括原始对象、关系记录、评估/检索配置和密集/稀疏向量有效负载。它不提供PostgreSQL PITR，跨区域复制或计划保留。

存储和访问

使用专用的私有MinIO bucket，默认为‘ dpi -recovery ’。API凭证需要对文档和恢复桶进行读/写/列表/删除访问；这两个桶都不能是公共的。规范对象位于‘ archives/{tenantId}/{archiveId}/ ’下面。下载的ZIP文件是传输视图，而不是规范存储。

从保留的源对象加上序列化的记录/向量和并发存档净空量来调整存储大小。在规定的保存期后明确删除档案；V1.4.0没有保留调度器。

##操作流程

1. 打开知识库并选择**Recovery**。
2. 创建存档。当活动摄取、评估和稀疏迁移工作处于静止状态时，突变被阻塞。
3. 等待‘ COMPLETED ’；检查项目计数、字节数和清单校验和。重试失败。只有完整的档案才能下载。
4. 还原已完成的存档。目标在“恢复”时仍然隐藏；重试恢复相同的目标。
5. 只在“COMPLETED”之后点击目标链接。放弃应该删除的失败还原。

永远不要在票据或审计描述中放置对象密钥、向量、签名url、凭证或原始提供者失败。对恢复错误进行编辑。

##故障处理

-休眠超时：让活动的工作完成，诊断卡住的作业，重试相同的存档。
-容量超出：减少并发工作；仅在测量依赖项和内存负载之后才提高配置的限制。
—档案校验失败：不进行恢复。保留证据直到诊断完成。
—恢复验证失败：目标器保持隐藏状态。纠正依赖项并重试相同的作业，或者放弃它。
-部分放弃失败：当对象、记录或矢量残留存在时，不报告清理完成。

Flyway V15和V16使已完成的恢复作业和恢复项目证据级联，并显式删除存档。在依赖存档保留清理之前应用这两个迁移。

基础设施备份

应用程序恢复是一层。生产环境仍然需要独立的、经过测试的PostgreSQL备份，包括MinIO桶等和Milvus，并在主机外保留。通过秘密管理保存加密密钥和凭证，永远不要在排练工件中保存。
