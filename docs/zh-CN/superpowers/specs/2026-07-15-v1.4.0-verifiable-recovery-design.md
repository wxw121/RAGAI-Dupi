# V1.4.0可验证恢复设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-15-v1.4.0-verifiable-recovery-design.md)





> **中文说明：** 本设计定义单个知识库的可验证归档与幂等恢复能力，覆盖对象存储、关系数据、评估与检索配置、向量数据、逐项校验和及审计结果。

# #目标

V1.4.0为一个知识库添加了应用程序级、可验证的备份和恢复。操作符可以归档知识库配置、原始MinIO对象、PostgreSQL记录、评估资产、检索配置和Milvus向量，然后将它们恢复到具有项目级校验和和可审计结果的新知识库中。

这弥补了V1.2 JSON导出留下的空白。导出仍然是轻量级的元数据迁移格式；它不是一个灾难恢复工件。

##释放顺序

V1.4作为两个独立可测试的增量交付：

1. V1.4.0：可验证的存档和还原。
2. V1.4.1：租户/用户上传配额、取消和终端故障通知。

只有V1.4.0适用于此设计及其实现计划。这两个增量可以共享操作UI约定、权限、审计基础设施和发件箱交付，但它们不共享状态机或数据库表。

# #范围

存档捕获：

-重构行为所需的知识库配置和生命周期元数据；
-来自MinIO的原始文档字节加上文档和规范块记录；
-修复知识库所需的RAG评估案例、质量方针和不可变检索概要；
-密集和活跃的稀疏Milvus矢量记录，包括检索所需的标量有效载荷；
-包含计数、字节大小、SHA-256校验和、嵌入模型和维度、集合模式、度量和索引设置的版本清单。

恢复的知识库接收到一个新的标识符，并且在验证成功之前一直不可用。记录源标识符是为了可追溯性，但永远不会作为目标主键重用。

# #非目标

—PostgreSQL时间点恢复或跨区域复制。
—外部备份产品的基础设施级卷快照或业务流程。
—恢复到、覆盖或修改现有知识库。
—对聊天会话、用户帐号、角色、审计历史、瞬态摄取任务、瞬态迁移任务进行归档。
—自动定时保留策略。V1.4.0支持显式删除存档；可以在测量生产使用情况后添加调度。
-将多个知识库合并到一个存档中。
—上传配额和取消，属于V1.4.1版本。

##选择的存储模型

恢复数据存储在与文档桶单独配置的专用MinIO私有桶中。默认桶名为“dpi -recovery”。API凭据必须能够访问这两个桶；这两个桶都不是公共的。

每个存档都使用这个前缀：

```text

archives/{tenantId}/{archiveId}/
  manifest.json
  records/knowledge-base.json
  records/documents.ndjson
  records/chunks.ndjson
  records/evaluation-cases.ndjson
  records/quality-policy.json
  records/retrieval-profiles.ndjson
  objects/{documentId}/{encodedFileName}
  vectors/dense.ndjson
  vectors/sparse.ndjson

```

项目是单独编写的，而不是作为一个ZIP或TAR。这允许有限内存流、项目级重试和校验和验证，而无需重写大的包。的清单。Json '最后写入。它本身的存在是不够的：它的数据库状态、模式版本、项目列表、计数和校验和也必须在可恢复存档之前进行验证。

操作人员可以通过经过身份验证的端点下载完整的归档文件作为流压缩文件。ZIP是从密封的前缀组装而来的，没有在API内存中缓冲整个包。这只是一个交通视角；对象集和已验证的清单仍然是规范存储表示。

领域模型

###恢复归档

‘ recovery_archives ’记录存档ID、租户ID、源知识库ID、状态、架构版本、存储桶和前缀、捕获的知识库修订、项目/计数摘要、错误代码和编辑错误消息、创建者、时间戳和清单校验和。

状态是:

```text

PREPARING -> CAPTURING -> VERIFYING -> COMPLETED
      |           |            |
      +-----------+------------+-> FAILED

```

“失败”档案仍然可诊断。重试在验证已写入的项目后恢复相同的归档ID；它不会创建另一个逻辑归档。操作符可以显式地删除失败或完成的存档。删除记录审计事件并幂等地删除前缀。

###恢复项目

‘ recovery_archive_items ’存储一个稳定的项目键、类型、源标识符、对象键、预期字节大小、SHA-256、项目状态、尝试计数和最后编校错误。唯一键‘ (archive_id, item_key) ’使捕获重试安全。

项目状态为“PENDING”、“WRITTEN”、“VERIFIED”和“FAILED”。在决定是否重用或重写现有对象之前，重试将验证该对象。

###恢复任务

“recovery_restore_jobs”记录归档ID、租户ID、一个目标知识库ID、状态、进度计数、错误信息、参与者和时间戳。

状态是:

```text

VALIDATING -> RESTORING_OBJECTS -> RESTORING_RECORDS
           -> RESTORING_VECTORS -> VERIFYING -> COMPLETED
                    |                   |
                    +-------------------+-> FAILED

```

当验证成功时，将分配一次目标知识库。重试失败的作业继续使用相同的目标。目标被标记为“正在恢复”，并被排除在正常的列表、检索、上传、重新索引、迁移、评估和聊天操作之外，直到作业达到“完成”。

‘ recovery_restore_items ’将每个存档项映射到其目标标识符和状态。‘ (restore_job_id, archive_item_id) ’是唯一的。对象写入、记录插入和向量插入使用确定性目标键，因此重复的步骤验证或替换相同的项，而不是复制它。

一致的归档捕获

档案创建获取现有知识库维护边界，并记录源修订。它等待活动摄取、重新索引、求值和稀疏迁移工作以达到终端状态。当捕获处于活动状态时，将使用结构化维护响应拒绝新文档突变、检索概要文件激活、质量策略突变、评估案例突变、重新索引和迁移转换。

如果知识库不能在配置的超时时间内变为静止状态，则在“捕获”之前创建存档失败。实现绝不能将混合活动摄取快照标记为可恢复快照。

静止后:

1. 快照PostgreSQL记录下的可读事务和可写记录项。
2. 在计算SHA-256时，将每个引用的原始对象流式传输到恢复桶。
3. 在密集和活跃的稀疏Milvus数据之间进行分页，并在不将整个集合加载到内存中的情况下写入向量项。
4. 根据持久项证据构建清单。
5. 重新阅读并核对每一项，然后编写并核对最终的清单。
6. 将存档标记为“COMPLETED”并释放维护边界。

任何故障都将存档标记为“FAILED”，保留项证据，并释放维护边界。通过api和审计日志暴露的错误被编校，并且不包括凭据、签名url、原始向量或提供者配置。

##恢复和验证

验证检查清单架构版本、清单校验和、租户所有权、所有项目校验和和大小、集合架构、度量/索引配置、嵌入模型和维度，以及声明的计数。验证在目标知识库可见之前完成。

恢复:

1. 使用新ID和冲突安全恢复的名称创建处于“正在恢复”状态的目标知识库。
2. 使用确定的目标对象键恢复原始对象。
3. 通过强制恢复等幂的特定于恢复的服务插入重新映射的文档、块、评估、策略和检索概要文件记录。
4. 在插入重新映射的向量之前，创建或验证版本化的密集和稀疏Milvus模式。
5. 验证MinIO字节、PostgreSQL计数和关系所有权、向量计数和有效负载校验和、嵌入兼容性和代表性检索行为。
6. 只有在每次验证通过后，才将目标标记为“就绪”，并将恢复作业标记为“已完成”。

失败的还原将保持隐藏并可重试。显式放弃通过补偿、幂等清除来删除隐藏目标，并记录一个审计事件。故障时不会运行自动清除，因为必须保留证据以供诊断。

## API和权限

所有路由的租户作用域都低于‘ /api/v1/知识库’或‘ /api/v1/ops/recovery ’，并且需要正常的资源授权加上新的‘ KB_RECOVERY ’权限。

最小API面：

-创建、列出、检查、重试、下载和删除档案；
—创建、列出、检查、重试和放弃恢复任务；
-揭露项目进度和修订的验证失败；
-拒绝跨租户存档并恢复访问，无论提供的标识符是什么。

每次创建、重试、完成、验证失败、删除和放弃都会发出一个审计事件。存档下载使用经过身份验证的流，并经过审计；API不返回永久的公共url或预先指定的url。

## Web体验

知识库操作区域获得一个“恢复”部分，该部分遵循现有的紧凑表、对话框、徽章和吐司模式。它显示存档状态、捕获时间、大小、计数、清单校验和、参与者和可用操作。Restore是一个确认流，声明将创建一个新的知识库。

进度轮询不能调整布局的大小。失败的项目显示已编辑的原因和重试操作。只有验证完成后，恢复的知识库才会出现在正常导航中。全局操作页面可能链接到失败的恢复作业，但不会复制完整的工作流。

##配置和操作

新的配置包括恢复桶名、静默超时、项目页面大小和最大并发归档/恢复作业。对于单节点撰写部署，默认值必须是保守的。Startup确保私有恢复桶存在，但不将其设为公共。

runbook记录了桶访问策略、存储大小、存档下载、恢复演练、故障诊断、显式删除，以及应用程序恢复和基础设施备份之间的区别。生产部署仍然需要独立的PostgreSQL、MinIO等和Milvus基础设施备份。

##故障处理

-跨系统写不是全局事务性的。稳定的项目键、持久的进度、确定的目标键和补偿使重试安全。
—在恢复写入开始之前，丢失或更改的归档对象未通过验证。
—通过版本比较检测到的源突变竞争不通过存档验证。
-矢量模式、模型、维度、计数或有效负载不匹配阻碍准备。
-进程重启后，非终端作业可以通过显式重试恢复；没有第二个作业可以同时操作同一存档或目标。
删除和放弃是幂等的，将部分清理报告为可诊断的失败，而不是声称成功。

测试和发布门

单元和服务测试涵盖状态转换、租户隔离、权限检查、稳定项键、校验和验证、编校、重试和补偿。控制器测试涵盖结构化错误和授权。

集成覆盖使用真正的PostgreSQL， MinIO和Milvus：

1. 创建包含原始对象、块、评估资产、密集向量和稀疏向量的夹具知识库；
2. 存档并恢复；
3. 比较对象SHA-256值、关系计数和所有权、向量计数和有效载荷校验和；
4. 针对源和目标运行相同的检索案例；
5. 注入一个中期捕获和中期恢复失败，重试，并证明没有创建重复的目标、对象、记录或向量；
6. 损坏一个存档项目，并在目标准备就绪之前证明恢复已被阻止。

浏览器E2E涵盖了存档创建、进度、完成的详细信息、恢复确认、恢复的知识库可见性和验证失败。发布门还运行API、Worker、Web、产品构建、Compose验证、秘密扫描和“git diff—check”。

直到真正的恢复排练工件记录了持续时间、项目计数、字节计数、校验和、检索比较和非平凡fixture知识库的清理结果之后，V1.4.0才算完成。

V1.4.1边界

下一个增量引入了状态为‘ PENDING ’、‘ COMMITTED ’和‘ RELEASED ’的持久配额保留；每个租户和每个用户保留的字节/文档限制；接受上传窗口计费；每个文件的幂等键；浏览器取消;stale-reservation和解;以及通过现有发件箱边界发送终端摄取失败通知。

V1.4.0不能预先创建配额表或将配额计费混合到恢复作业中。恢复写操作是特权恢复操作，不消耗上传窗口配额；在成功恢复之后，当引入增量时，保留的文档和字节使用将被调和到V1.4.1记帐中。
