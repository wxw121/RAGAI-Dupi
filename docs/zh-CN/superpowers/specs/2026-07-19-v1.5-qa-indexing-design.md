# V1.5 QA索引设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-19-v1.5-qa-indexing-design.md)





> **中文说明：** 本设计在不改变经典与父子检索默认行为的前提下增加 QA 辅助索引，通过现有 API 大模型生成并校验问答候选，同时保留来源追踪信息。

# #总结

添加P1 qa辅助的索引路径，而不更改现有的经典或父-子默认值。在摄取期间，Worker将源块发送到内部API端点。API重用配置的‘ LlmClient ’来生成经过验证的问题/答案候选人。Worker将候选对象转换为QA块，将它们与源来源一起存储，并将它们索引在正常检索条目旁边。

# #目标

-为每个源块生成一个小的、有限的候选问题和答案集。
-保存从QA块到源块、文档和父上下文的出处。
-使“QA_ASSISTED”检索QA条目和原始条目。
-在将源上下文返回给调用者时，使‘ COMBINED ’检索父子和QA条目。
-保持QA生成失败非致命：原始块仍然可搜索。
保持“CLASSIC”和现有的“VECTOR/HYBRID/RERANK”行为不变。

# #非目标

—不自动切换默认配置文件。
- Worker中没有特定于提供商的LLM配置。
-查询时不生成答案；QA条目仅是检索快捷键。
-在P1批次中没有分数权重调整。

##选择的架构

API QA生成端点

添加一个内部端点：

POST / api / v1 /内部知识库/ {kbId} / qa-candidates '

要求:

```json

{
  "docId": "uuid",
  "sources": [
    {"chunkId": "uuid", "content": "...", "metadata": {}}
  ]
}

```

回应:

```json

{
  "candidates": [
    {
      "sourceChunkId": "uuid",
      "question": "...",
      "answer": "..."
    }
  ]
}

```

API验证所请求的文档是否属于知识库，并通过现有的LlmClient发送严格的JSON提示。，解析和验证候选人，删除重复的问题，并限制每个来源的候选人数量。当QA生成运行时，源块还没有持久化，因此它们的id /内容仅在经过身份验证的Worker/内部密钥边界上可信。

工人摄取流

-“CLASSIC”：不变。
- ' PARENT_CHILD '：与P0不变；父节点被持久化，子节点被索引。
- ' QA_ASSISTED '：正常块被持久化/索引；普通块作为QA源发送；生成的QA块被持久化/索引。
—“COMBINED”：生成父节点和子节点，如P0；家长作为QA来源；生成的QA块被保存/索引在子块旁边。

每个QA块都使用“问题：……”\ nAnswer:…’和元数据：

```json

{
  "chunk_role": "qa",
  "source_chunk_id": "uuid",
  "qa_question": "...",
  "qa_answer": "...",
  "qa_source_kind": "original_chunk"
}

```

QA块接收它自己的UUID和块索引。原始源块对于引用和答案上下文保持权威性。

失败和重试行为

如果内部调用、LLM响应、JSON解析或候选验证失败，Worker将记录失败并完成原始块的摄取。没有格式化的QA块被持久化或索引。Reindex/retry再次运行QA生成，因此可以在不单独迁移的情况下恢复瞬态提供程序故障。

检索行为

对于‘ QA_ASSISTED ’和‘ COMBINED ’，元数据包含‘ source_chunk_id ’的命中将扩展到源块。响应保留QA来源字段：

——“matched_qa_chunk_id”
——“qa_source_chunk_id”
——“qa_question”
——“qa_answer”
——“概要”

对于“COMBINED”，当被解析的源是子源时，在QA源解析之后应用父展开。返回的内容始终是源/父上下文，而元数据标识导致匹配的QA或子条目。拒绝跨kb和跨文档的源引用。

Hybrid BM25继续排除仅父块作为直接条目，但是QA块仍然是合格的条目。

# #测试

- API：候选提示响应解析，验证/重复数据删除/封顶，错误响应失败，端点委托和KB/文档边界。
- Worker: QA候选‘ TextChunk ’转换，来源元数据，QA辅助和组合源选择，非致命生成回退，仅索引可搜索条目。
-检索：QA hit扩展，结合QA + parent扩展，出处保存，交叉文件拒绝。
-现有的经典/亲子/混合测试保持绿色。

##推出门

保持默认配置文件为“CLASSIC”。只有在对候选配置文件物理构建的索引进行评估后，才能启用每个知识库的“QA_ASSISTED”或“COMBINED”，并且在命中率和引用质量方面没有回归。当前的多配置文件运行比较一个活动物理索引上的检索行为；配置文件隔离的索引（或可过滤的超集）和自动默认启用的策略仍然是P2。
