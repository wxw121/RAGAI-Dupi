# V1.1可观察性和评估设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-12-v1.1-observability-evaluation-design.md)





> **中文说明：** 本设计通过真实浏览器端到端门禁、摄取与索引诊断、轻量 RAG 评估闭环以及上传/运维可见性，让现有 V1 系统更易测试、诊断和运营。

# #总结

V1.1的重点是使现有的V1系统可测试、可诊断和更易于操作。目标不是广泛的平台重写。目标是四个闭环：

-真正的浏览器端到端门户登录和核心UI流程；
-摄取/索引诊断失败或停滞的文件解释发生了什么和下一步怎么做；
- RAG评估，以便从脚本和产品表面检查检索质量；
P1首先将治理和运维警报作为可见性上传，而不会在此版本中添加新的硬配额阻塞。

当前api级别的E2E脚本和RAG脚本仍然有用，但它们不能证明用户可以在浏览器中操作产品。当前的摄取和警报数据是存在的，但其中有太多隐藏在原始字段或分散的面板中。V1.1将这些功能转换为显式门和可见的操作员反馈。

##考虑的方法

方法A：仅ci质量门

只添加剧作家和RAG脚本，不更改产品UI。

优点:
—最低代码面。
-快速添加和易于在CI中运行。

缺点:
-不能帮助管理员诊断用户摄取失败。
- RAG评估对非工程师是不可见的。
-上传管理和警报仍然分散。

方法B：全可观测子系统

添加持久的评估运行、警报规则、通知通道、上传配额和完整的操作指示板。

优点:
—长期可扩展方向。
-明确评估和警报的审计跟踪。

缺点:
-对于小版本来说太大了。
-在产品有足够的运行历史之前引入新的模式和警报语义。
-有延迟P0浏览器闸门和摄入诊断的风险。

方法C：现有流上的薄封闭层

添加剧作家浏览器E2E，用计算诊断丰富现有的摄取作业响应，添加重用“/retrieve”的轻量级RAG评估选项卡，并公开当前的治理阈值和可操作的操作警报。

优点:
-使用当前架构和现有服务。
-使最痛苦的失败模式立即可见。
为工程师和管理员提供可重复的质量检查。
将P1上传治理作为可见性和警报，而不是一个新的策略引擎。

缺点:
-评估报告是轻量级的，历史上还没有持久化。
—上传配额是以后的工作。
-警报传递仍然只在产品/API中，而不是webhook/email/IM。

选择方法：方法C。

# #范围

### P0：真正的浏览器E2E门

在“services/web”下添加剧作家和一个根PowerShell包装器：

-通过“http://localhost:8080”的真实页面登录。
-使用cookie + CSRF行为，就像产品使用它一样。
—通过UI创建知识库。
—进入KB详细信息页面。
-验证文档，聊天，ops审计，帐户和角色页面渲染没有控制台错误。
—当本地堆栈配置完成且稳定时，可以包含上传流；否则，第一个门仍然应该证明经过身份验证的导航和表单流。

凭据由环境变量提供：

——“E2E_ADMIN_USERNAME”
——“E2E_ADMIN_PASSWORD”
- ‘ E2E_BASE_URL ‘，默认’ http://localhost:8080 ’

如果凭据丢失，大门将失败并给出明确的信息。它不能依赖于本地打开模式或旁路身份验证。

### P0：摄取/索引诊断关闭

使用计算诊断对象扩展现有的摄取作业响应。不需要模式迁移。

诊断领域:

- ' severity ': ' info ‘, ‘ warning ’或’ error '；
-“总结”：一个简短的面向操作员的解释；
- ' nextAction '：具体操作，如等待，重试，重新索引，检查嵌入配置，或检查存储/工作；
- ' retryable '：手动重试是否合适；
-“停滞不前”：工作是否没有更新到一个实际的阈值；
- ‘ agesseconds ‘和’ lastUpdatedSeconds ’。

诊断由现有字段计算：

-文件状态；
-工作状态；
-工作阶段；
-重试计数；
-错误信息；
——时间戳。

UI变化:

—文档表显示最新的作业状态和简明的诊断行。
-索引维护表显示诊断，不只是原始错误文本。
失败/死信作业像以前一样暴露重试，但现在解释重试是否可能有用。
-在不等待硬故障的情况下调用暂停的挂起/处理作业。

### P0: RAG评估闭包

保留现有的“示例/粗略评估案例”。’和‘scripts/rag-regression-eval’。Ps1 '，但使评估可见和可重用：

-在KB详细信息中添加一个“RAG。
—页面使用‘ /retrieve ’对当前KB运行相同的检查样式。
-显示每个案例的通过/失败，命中计数，期望的文件，匹配的令牌，检索模式，回退原因和嵌入维度。
-为‘ /retrieve ’和共享eval case/result类型添加前端API包装器。
-更新脚本输出，以便每个案例都有足够的细节来从生成的JSON报告中诊断故障。

此版本不保留评估历史记录。最近的页面运行足以作为操作员的反馈，而脚本报告足以作为gate证据。

### P1：上传治理和运维警报

这个版本暴露了当前的操作护栏，并增加了有用的产品内警报。它没有添加新的硬配额引擎。

添加元数据：

上传速率限制启用/请求/窗口；
最大等待任务；
-摄取恢复最大尝试；
—审计警报窗口和失败阈值；
- multipart最大文件大小（如果可用）。

UI变化:

—上传区域始终显示支持的文件类型。
—如果当前调用方可以加载ops元数据，则上传区域还会显示配置的速率限制和队列提示。
—Ops审计页面显示当前的护栏设置和活动警报。
-警报列表包括审计失败峰值、摄取死信/失败作业和矢量清理失败（现有存储库可以支持计数）。

未来的配额执行仍然不在这个版本的范围之内。

##产品行为

浏览器门

闸门由：

```powershell

powershell -NoProfile -ExecutionPolicy Bypass -File scripts/e2e-browser-gate.ps1

```

它从一个干净的浏览器上下文开始，登录，然后失败：

-证件缺失；
-登录错误；
-导航超时；
-在检查流上可见的错误吐司；
未捕获的页面异常；
-来自应用程序代码的浏览器控制台错误。

摄入诊断

管理员应该能够回答文档页面上的三个问题：

1. 文件现在在做什么？
2. 如果失败了，为什么会失败呢？
3. 接下来我该做什么？

UI应该更喜欢简短的摘要而不是原始的堆栈样式的消息，同时在有用的时候仍然显示原始的错误。

### RAG评估

评估页面故意很简单：

-选择或使用内置机箱；
-点击运行；
-见通过/不通过表；
-检查检索诊断。

初始的内置案例镜像‘ examples/rag-eval-cases.json ’。自定义持久化用例管理不是V1.1的一部分。

### Ops警报

在V1.1中警报仍然是基于拉的。产品在ops页面和API中公开它们。外部通知集成被延迟。

——“IngestDiagnosisResponse”
- ' IngestJobResponse '的可选字段：
——“documentFileName”
——“documentStatus”
——“诊断”

现有端点仍然存在：

-“GET /api/v1/knowledge-bases/{kbId}/ ing- jobs”
-“GET /api/v1/知识库/{kbId}/documents/{docId}/ ing- job”

检索API客户端

##开放风险

-完全浏览器端到端依赖于运行的本地堆栈和真实的管理员帐户。当先决条件缺失时，门必须清楚地失败。
- RAG评估质量取决于当前KB内容。内置示例用例只有在上传示例文档或用户计算等效的KB时才稳定。
-一些项目文本在PowerShell输出中呈现mojibake，但源文件被视为UTF-8。编辑应该保留UTF-8。
