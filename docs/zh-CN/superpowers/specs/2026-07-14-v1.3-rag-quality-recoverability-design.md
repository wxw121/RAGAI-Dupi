# V1.3 RAG Quality and Recoverability Design / RAG 质量与可恢复性设计

[English](../../../en/superpowers/specs/2026-07-14-v1.3-rag-quality-recoverability-design.md)





## Goal / 目标

V1.3使检索质量在更改生产检索行为之前可测量和强制执行。然后，它使混合检索可观察和生产就绪，然后是上传治理和可验证的恢复工作流。

V1.3 在改变生产检索行为前，将检索质量变为可度量、可比较、可阻断的信号；随后将混合检索升级为可观测、可用于生产的能力，并完成上传治理和可校验的恢复流程。

## Scope and Delivery Order / 范围与交付顺序

1. **P0-1: RAG quality gate / RAG 质量门禁**
2. **P0-2: Production hybrid retrieval tuning / 混合检索生产调优**
3. **P1-1: Upload governance / 上传治理**
4. **P1-2: Verifiable disaster recovery / 可校验完整灾备恢复**

在这个版本中，灾难恢复仍然是P1。在依赖于应用程序管理的恢复的生产启动之前，必须将其提升到P0。

本版本灾备恢复保持 P1。若后续生产上线依赖应用侧恢复能力，必须在上线前提升为 P0。

## Current Constraints / 当前约束

- RAG评估保留案例、运行和每个案例的结果，但不存储提升的基线、趋势度量或阻塞决策。
-混合检索目前调用Worker，每个请求获取完整的知识库语料库，并在RRF和可选的重新排序之前构建内存中的“BM25Okapi”索引。这不是一个生产稀疏索引。
—现有的导出/导入只恢复知识库配置和评估用例。它故意不恢复原始MinIO对象、文档记录、块或向量。
—上传控制暴露率和队列保护，但没有持久的租户/用户配额、客户端取消、上传失败事件或高级操作过滤器。

- 现有 RAG 评估已持久化用例、运行和逐用例结果，但没有已晋升基线、趋势指标或阻断结论。
- 现有混合检索由 Worker 在每次请求中获取整库语料、内存构建 `BM25Okapi`，再做 RRF 和可选重排；这不是生产级 sparse 索引。
- 现有导出/导入只恢复知识库配置和评估用例，刻意不恢复 MinIO 原文件、文档记录、分块或向量。
- 上传控制目前只展示限流和队列护栏，没有持久化租户/用户配额、客户端取消、上传失败事件或高级运维筛选。

## P0-1: Quality Gate / 质量门禁

### Model / 数据模型

每个知识库都有一个可变的质量策略和零个或一个活动基线。基线指向一个不可变的已完成的计算运行；它由授权的管理员显式提升，并且永远不会自动更改。

每个知识库拥有一份可修改的质量策略和零或一个当前基线。基线指向一次不可变的已完成评估运行，只能由具备权限的管理员显式晋升，绝不自动替换。

该政策包括：

‘ minimumPassRate ’：绝对通过率下限。
- ' maximumPassRateDrop '：相对于基线的最大允许下降。
- ' maximumNewFailures '：先前通过的基线案例允许失败的数量。
- ' blockwhenunbaselines '：是否闸门执行没有基线阻塞。

策略包含：

- `minimumPassRate`：绝对通过率下限。
- `maximumPassRateDrop`：相对基线允许的最大通过率下降。
- `maximumNewFailures`：基线中通过、当前失败的用例数上限。
- `blockWhenUnbaselined`：无基线运行是否阻断。

每次完成的运行都会记录不可变的汇总指标：通过率、合格的预期文件命中率、合格的关键字命中率、平均命中计数、回退计数和延迟百分比汇总。每个结果接收一个比较状态：“改进”、“不变”、“回归”或“新的”。

每次完成运行记录不可变汇总指标：通过率、适用用例的期望文件命中率、适用用例的关键词命中率、平均命中数、降级次数和延迟分位摘要。每条结果具备 `IMPROVED`、`UNCHANGED`、`REGRESSED` 或 `NEW` 对比状态。

### Gate Decision / 门禁判定

服务返回‘ PASS ’、‘ WARN ’、‘ BLOCKED ’或‘ unbaselines ’。当运行违反绝对阈值、超过基准通过率下降或超过新故障允许时，运行被“阻塞”。‘ WARN ’仅用于诊断，不会产生非零的进程退出代码。

服务返回 `PASS`、`WARN`、`BLOCKED` 或 `UNBASELINED`。违反绝对阈值、超过基线通过率降幅或新增失败数上限时为 `BLOCKED`。`WARN` 仅用于诊断，不产生非零退出码。

'脚本/ rag-quality-gate。ps1 ‘运行一个评估，只对’ BLOCKED ‘和策略配置的’ unbaseline '退出非零。它是CI/发布集成点；正常的UI运行仍然返回完整的诊断结果。

`scripts/rag-quality-gate.ps1` 运行评估，并仅在 `BLOCKED` 或策略规定需阻断的 `UNBASELINED` 时返回非零退出码。该脚本是 CI/发布接入点；普通 UI 运行仍返回完整诊断结果。

### Deterministic Comparison Rules / 确定性对比规则

‘ caseKey ’是稳定的比较恒等式。运行还存储规范化查询和每个断言字段的指纹。缺少基线密钥或指纹发生变化的当前病例为“NEW”；仅删除基线的案例将单独报告，并且永远不会被视为新的失败。只有匹配的密钥和指纹对才有助于回归度量。

`caseKey` 是稳定的对比标识。每次运行还保存规范化问题和全部断言字段的指纹。当前用例缺少基线键或指纹变化时为 `NEW`；只存在于基线的已删除用例单独报告，绝不计入新增失败。只有键和指纹都相同的用例对参与退化指标计算。

“BLOCKED”表示违反了绝对策略阈值、允许的通过率下降或允许的新失败计数；当‘ blockwhenunbaselines ’为true时，它也适用于非基线运行。“unbaselines”表示所有绝对阈值都已通过，但不存在活动基线，并且策略是非阻塞的。“WARN”表示没有阻塞规则失败，但当前和基线情况集不能完全比较。“通过”需要一个活跃的基线，完全的可比性，以及每个阻塞规则的通过。基线只能从完成的“PASS”运行中提升，或者从完成的“unbaseline”运行中提升，所有绝对阈值都通过以建立第一个基线；‘ WARN ’和‘ BLOCKED ’运行永远不会被提升。

`BLOCKED` 表示违反绝对策略阈值、允许通过率降幅或允许新增失败数；当 `blockWhenUnbaselined` 为真时，无基线运行也为 `BLOCKED`。`UNBASELINED` 表示绝对阈值均通过但没有活动基线，且策略不阻断。`WARN` 表示没有阻断规则失败，但当前与基线用例集不可完全比较。`PASS` 要求存在活动基线、完全可比且所有阻断规则通过。只有已完成的 `PASS` 运行，或用于建立首个基线且全部绝对阈值通过的已完成 `UNBASELINED` 运行可以被晋升；`WARN` 与 `BLOCKED` 运行永不可晋升。

### UI / 界面

RAG评估选项卡显示策略控制、活动基线、紧凑的度量趋势、gate决策和每个案例基线比较。提升基线需要明确的确认，并且只有在完成通过运行后才可用。

RAG 评估页展示策略控件、当前基线、紧凑的指标趋势、门禁结论及逐用例基线对比。仅已完成且通过的运行可被晋升为基线，操作需显式确认。

## P0-2: Production Hybrid Retrieval / 混合检索生产调优

### Alternatives and Decision / 方案比较与决策

1. 调优当前内存中每个请求的BM25实现。该方法费时少，但保留了全语料库提取，不能满足稀疏索引生成的要求。
2. **选择：将稀疏检索迁移到Milvus稀疏/BM25倒域**，在迁移期间具有临时的，可观察的Worker回退。它删除了每个请求的语料库构造，同时保留了当前的Milvus部署边界。
3. 添加OpenSearch / Elasticsearch。它更加灵活，但是为这个版本增加了不必要的生产依赖。

1. 调优当前每请求内存 BM25，实现成本低，但保留整库拉取，无法满足生产 sparse 索引要求。
2. **选定：迁移到 Milvus sparse/BM25 倒排字段**，并在迁移期间保留可观测的 Worker 回退。该方案移除每请求语料构建，同时维持当前 Milvus 部署边界。
3. 引入 OpenSearch/Elasticsearch，灵活性更高，但为本版本引入了不必要的生产依赖。

### Retrieval Profile / 检索配置档案

检索配置作为命名概要文件进行版本控制。它捕获向量候选计数、稀疏候选计数、RRF常数、稀疏索引/搜索参数、重新排序启用、重新排序候选限制和最终top-K。运行时会保留完整的概要文件快照，而不仅仅是一个布尔重新排序标志。

检索配置以命名、可版本化的档案保存，包含向量候选数、sparse 候选数、RRF 常量、sparse 索引/搜索参数、是否启用 Rerank、Rerank 候选上限和最终 top-K。评估运行保存完整档案快照，而不只保存 Rerank 布尔值。

每个知识库都有一个活动的生产概要文件。只有当完成的质量门运行使用其精确的不可变快照并返回“PASS”时，配置文件才能被激活。激活被审计，并立即使API将该配置文件传递给Worker。回滚是对先前通过的概要文件的审计激活；它从不改变历史运行。

每个知识库恰有一份活动生产配置档案。只有当已完成的质量门禁运行使用该档案的精确不可变快照且返回 `PASS` 时，档案才能启用。启用操作会被审计，并立即让 API 将该档案传递给 Worker。回滚是对先前通过档案的一次审计启用，绝不修改历史运行。

检索响应添加了阶段解释：向量候选秩、稀疏候选秩、融合分数/秩、重新排序分数/秩、应用概要文件版本和任何回退原因。原始向量和提供者凭证永远不会暴露。

检索响应增加阶段解释：向量候选排名、sparse 候选排名、融合分数/排名、重排分数/排名、应用的配置档案版本和任何回退原因。不得暴露原始向量或供应商凭据。

### Benchmark / 压测与对比

基准脚本通过“VECTOR”、“HYBRID”和“HYBRID+RERANK”配置文件运行相同的基线数据集。它报告质量门度量、P50/P95延迟、回退率、语料库大小、概要快照和每个案例的等级变化。UI使用这些不可变快照比较两个运行或配置文件。

压测脚本以同一基线数据集运行 `VECTOR`、`HYBRID` 和 `HYBRID+RERANK` 配置，报告质量门禁指标、P50/P95 延迟、回退比例、语料规模、配置快照和逐用例排名变化。UI 基于这些不可变快照比较任意两次运行或两个配置档案。

### Sparse Migration / Sparse 迁移

迁移使用新版本的Milvus集合/模式，而不是修改现有的活动集合。它有明确的状态：‘准备’，‘回填’，‘ DUAL_WRITING ’， ‘ shadow_validation ’， ‘ CUTOVER ’， ‘ COMPLETED ’和‘ FAILED ’。Backfill读取规范的PostgreSQL数据块，验证嵌入的维度和稀疏的模式/索引参数，并幂等地写入新的集合。新的和重新索引的文档双重写入，直到切换。

迁移使用新的版本化 Milvus collection/schema，而不就地修改在线集合。状态明确为 `PREPARING`、`BACKFILLING`、`DUAL_WRITING`、`SHADOW_VALIDATING`、`CUTOVER`、`COMPLETED` 和 `FAILED`。回填读取 PostgreSQL 中的规范分块，校验 embedding 维度及 sparse schema/索引参数，并幂等写入新集合。新上传和重建索引的文档在切换前双写。

割接需要100%的预期块覆盖率、匹配的尺寸、候选概要文件没有阻塞的质量门结果、P95延迟不低于当前生产基线的1.25倍，并且没有更高的回退率。遗留Worker ‘ BM25Okapi ’路由仅在双写和影子验证期间保留在审计的功能标志后面；在满足这些验收标准的稳定发布之后，它被禁用，但是保留的代码直到以后的发布才被删除。

切换要求预期分块覆盖率 100%、维度一致、候选档案没有被质量门禁阻断、P95 延迟不高于当前生产基线的 1.25 倍，且回退率不升高。旧 Worker `BM25Okapi` 路径仅在双写和影子验证期间通过可审计特性开关保留；满足这些验收条件并稳定一个版本后禁用，但代码留待后续版本删除。

## P1-1: Upload Governance / 上传治理

### Quotas and Cancellation / 配额与取消

持久配额记帐强制对每个租户和每个用户在一个时间窗口内保留的字节、保留的文档计数和上传字节进行限制。上传许可自动预留对象存储前的容量；失败和删除会释放保留。现有的利率限制仍然是一种单独的短窗口保护。

持久化配额核算对租户和用户分别执行已保留字节数、已保留文档数和时间窗口上传字节数限制。上传准入在写入对象存储前原子预占容量；失败和删除释放预占。现有限流继续作为独立的短窗口保护。

Web批处理上传器使用‘ AbortController ’。取消停止正在运行的HTTP请求，并防止批处理中未启动的文件；已经接受的文档仍然是可审计的，并继续进行正常的摄取处理。

Web 批量上传使用 `AbortController`。取消会中止正在传输的 HTTP 请求并阻止批次中未开始的文件；已被服务端接收的文档保留可审计状态，并继续走正常摄入处理。

每个文件都带有一个等幂键。存储/文档保留有‘ PENDING ’， ‘ COMMITTED ’和‘ RELEASED ’状态。保留的字节和保留的文档容量只有在对象和文档记录被接受后才会提交，然后在终端存储失败或删除时释放。窗口上传字节衡量接受的上传尝试，并且在服务器接受后不释放，防止取消/重试配额绕过；使用相同的等幂键进行重试不会收取两次费用。在验证对象和文档都未被接受后，调度的协调程序释放过时的“PENDING”保留。

每个文件携带幂等键。存储/文档预占具有 `PENDING`、`COMMITTED` 和 `RELEASED` 状态。已保留字节和已保留文档容量只在对象与文档记录被接收后提交，并在最终存储失败或删除时释放。时间窗口上传字节按服务端已接收的上传尝试计量，接收后不因取消/重试释放，以防绕过配额；相同幂等键的重试不重复计费。定时协调器在确认对象和文档均未被接收后释放过期 `PENDING` 预占。

### Notifications and Operations / 通知与运维

上传和摄取终端失败将创建发件箱通知事件，并使用现有的Webhook交付边界。操作过滤添加租户、参与者、知识库、文档、状态、阶段和时间范围，而不暴露跨租户数据。

上传和摄入的最终失败创建 outbox 通知事件，并复用现有 Webhook 投递边界。运维筛选增加租户、操作者、知识库、文档、状态、阶段和时间范围，同时不得暴露跨租户数据。

## P1-2: Verifiable Disaster Recovery / 可校验完整灾备恢复

### Archive / 归档

版本化的恢复存档包含清单、原始MinIO对象、知识库配置、文档和块记录、评估资产、矢量记录和SHA-256校验和。归档与V1.2元数据导出不同，并带有显式的恢复模式版本。

版本化恢复归档包含清单、MinIO 原始对象、知识库配置、文档和分块记录、评估资产、向量记录及 SHA-256 校验值。该归档不同于 V1.2 元数据导出，并带有明确的恢复 schema 版本。

### Restore State Machine / 恢复状态机

还原总是创建一个新的知识库，并通过“validation”、“RESTORING_OBJECTS”、“RESTORING_RECORDS”、“RESTORING_VECTORS”、“VERIFYING”、“COMPLETED”或“FAILED”进行。它验证清单校验和、对象字节、文档/块计数、向量计数、向量维度和向量有效负载校验和。失败的还原仍然是可诊断和可重试的，但永远不会公开为就绪。

恢复始终创建新知识库，并经过 `VALIDATING`、`RESTORING_OBJECTS`、`RESTORING_RECORDS`、`RESTORING_VECTORS`、`VERIFYING`、`COMPLETED` 或 `FAILED` 状态。流程校验清单校验和、对象字节、文档/分块数量、向量数量、向量维度和向量载荷校验和。失败恢复保持可诊断、可重试，但绝不作为可用知识库暴露。

存档创建获得知识库维护锁，等待活动的摄取/重新索引工作完成，并拒绝新的突变，直到清单被密封。清单记录捕获的修订、源集合模式、度量/索引设置、嵌入模型和维度，以及每个对象/向量的校验和。当无法达到安静时，档案被拒绝；它从不将混合活动摄取快照表示为可恢复的。

创建归档时会获取知识库维护锁，等待活动摄入/重建任务完成，并在清单封存前拒绝新的变更。清单记录捕获版本、源 collection schema、度量/索引设置、embedding 模型和维度，以及每个对象/向量的校验和。无法达到静止状态时归档直接失败；绝不将混合活动摄入快照标记为可恢复。

一个恢复作业一次创建一个目标知识库。在补偿不完整的项目后，重新尝试作业恢复相同的目标和存档项目清单；它永远不会创建另一个目标。在插入存档的矢量有效负载之前，矢量恢复提供或验证完全兼容的版本化的Milvus模式。模式、度量、索引、模型、维度、计数和有效负载-校验和不匹配块在目标准备就绪之前恢复。

恢复任务只创建一次目标知识库。任务重试会在补偿未完成项后，基于同一目标和归档项清单继续执行，绝不创建另一个目标。向量恢复会先创建或校验完全兼容的版本化 Milvus schema，再插入归档向量载荷。schema、度量、索引、模型、维度、数量或载荷校验和不匹配时，恢复被阻断，目标不得成为可用状态。

恢复操作使用专用权限并审计每个存档、验证失败、重试和已完成的恢复。跨系统写不能是全局事务性的；补偿记录和等幂的存档项键使重试安全。

恢复操作使用独立权限，并审计每次归档、校验失败、重试和完成恢复。跨系统写入无法形成全局事务；补偿记录和幂等归档项键确保重试安全。

## Non-Goals / 非目标

- V1.3中没有LLM回答质量判断；门只评估检索证据。
—没有自动提升质量基线。
—不依赖OpenSearch/Elasticsearch。
—无跨区域存储复制或时间点PostgreSQL恢复业务。
-没有静默部分恢复或源知识库标识符的直接重用。

- V1.3 不做 LLM 回答质量裁判；门禁只评估检索证据。
- 不自动晋升质量基线。
- 不引入 OpenSearch/Elasticsearch。
- 不做跨区域存储复制或 PostgreSQL 时间点恢复编排。
- 不允许静默部分恢复，也不直接复用源知识库标识。
