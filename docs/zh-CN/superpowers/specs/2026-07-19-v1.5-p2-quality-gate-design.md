# V1.5 P2剖面隔离评估和质量门设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-19-v1.5-p2-quality-gate-design.md)





> **中文说明：** 本设计确保不同检索配置基于等价源数据进行比较，并要求非 `CLASSIC` 配置在命中率与引用质量不退化后才能成为知识库默认配置。

# #总结

P2使检索配置文件在等效源数据之间的比较有效，并防止非经典配置文件成为知识库默认值，直到它证明点击率和引用质量不会回归到“经典”。

所选择的体系结构为每个文档构建一个可过滤的Milvus超集。它保留了现有的VECTOR/HYBRID/RERANK执行模式，将“CLASSIC”保留为初始默认值，为“COMBINED”添加确定性加权融合，并在RAG评估运行中存储修订绑定的门决策。

# #目标

-在一次摄取期间从同一文档版本构建所有四个检索配置文件。
-在排名前按配置文件筛选矢量和BM25候选人。
-将“CLASSIC”和候选配置文件与等效索引源数据进行比较。
结合子和QA路由明确，可诊断的权重。
-拒绝非经典的默认配置文件更改，除非当前评估门通过。
保持旧的经典集合可读，而现有的知识库重建v2配置文件索引。
保留非致命的QA生成回退和现有的检索执行模式。

# #非目标

—自动修改知识库默认配置文件，无需用户操作。
—在Web UI中添加每个知识库的分数调优控件。
-取代现有的VECTOR/HYBRID/RERANK执行模式。
-提供零成本索引；用户接受了更高的嵌入、QA生成和Milvus存储成本，以换取有效的A/B比较和即时配置文件切换。
-在每个知识库完成v2转换之前删除遗留的Milvus集合。

##选择的架构

Filterable超集

创建一个名为“${MILVUS_COLLECTION}_profiles_v2”的新Milvus集合。不要尝试将标量字段添加到现有集合中。

每个索引记录包含现有字段加上：

- ' entry_kind ': ‘ original ‘, ‘ child ’或’ qa ’
- ' profile_classic '：布尔值
- ' profile_parent_child '：布尔值
- ' profile_qa_assisted ': Boolean
- ' profile_combined '：布尔值

PostgreSQL块元数据存储相同的条目类型和一个‘ profile_scope ’列表。父块仍然仅限postgresql，并且永远不会是直接向量或BM25项。

一次摄入产生：

|入口|持续|向量索引|配置文件范围|
|---|---:|---:|---|
|原始块| yes | yes | ' classic ', ' qa-assisted ' |
|父块|是|否| ' Parent -child ', ' combined ' |
|子块| yes | yes | ' parent-child ', ' combined ' |
| QA来自原始块| yes | yes | ' QA -assisted ' |
| QA来自父块| yes | yes | ' combined ' |

原始块被经典和qa辅助重用。子块被父-子重用并合并。QA条目保持独立，因为QA辅助和组合使用不同的源语义。

###配置文件过滤

配置文件过滤发生在排名之前：

- ' CLASSIC ': ‘ profile_classic == true ’
- ' PARENT_CHILD ': ‘ profile_parent_child == true AND entry_kind == "child" ’
- ' QA_ASSISTED ': ‘ profile_qa_assisted == true ’
- ‘ COMBINED ‘子路由：’ profile_combined == true AND entry_kind == "child" ’
- ‘ COMBINED ‘ QA路由：’ profile_combined == true AND entry_kind == " QA " ’

Worker在BM25评分之前应用等效的‘ profile_scope ’和‘ entry_kind ’过滤器。因此，请求级配置文件覆盖选择一个孤立的逻辑索引，而不是在一个未过滤的物理索引上重播不同的扩展行为。

组合加权融合

Combined独立检索子路由和QA路由，因此一条路由不会消耗另一条路由的候选预算。

使用加权倒数秩融合：

‘ routeScore = routeWeight / (rrfK + rank + 1) ’

默认值:

-儿童体重：“1.0”
QA权重：“0.8”
- RRF k: ‘ 60 ’

配置键由API和Worker通过环境支持的属性共享：

——“COMBINED_CHILD_WEIGHT”
——“COMBINED_QA_WEIGHT”
——“RRF_K”

所有权重必须为正，并且‘ RRF_K ’必须至少为1。矢量模式融合了子矢量和qa矢量。混合模式融合了child-vector、child- bm25、QA-vector和QA- bm25，为两个执行路径使用子或QA权重。可选择在融合后重新排序。检索诊断包括路由计数、有效权重和RRF K。

在QA源和父扩展之后，最终命中由最终块ID进行重复数据删除，并保留最高融合分数。

##索引版本和修订

Flyway V10增加：

——“文档。index_schema_version INTEGER NOT NULL DEFAULT 1 '
——“knowledge_bases。index_revision BIGINT NOT NULL DEFAULT 0 '
——“rag_eval_runs。index_revision BIGINT”
——“rag_eval_runs。gate_summary JSONB NOT NULL默认‘{}’：：JSONB '
——“rag_eval_run_results。hit_passed BOOLEAN NOT NULL默认FALSE
——“rag_eval_run_results。citation_eligible BOOLEAN NOT NULL默认FALSE
——“rag_eval_run_results。citation_passed BOOLEAN NOT NULL默认FALSE

目标文档索引架构版本为‘ 2 ’。

‘ index_revision ’随着索引知识的变化而变化：

-一个文档的摄取完成，它的块取代以前的版本；
-文件被删除；
-知识库重新索引启动；
-否则操作将使当前物理索引失效。

在执行用例之前，评估运行捕获知识库修订。只有当gate决策的运行版本等于概要文件更改时的知识库版本时，它才是当前的。

当使用模式版本2成功索引每个活动文档时，知识库概要索引就准备好了。一个空白的知识库是不可能打开大门的。

升级和切换

遗留集合和v2集合在rollout期间共存。

1. 在知识库准备就绪之前，只有“CLASSIC”可以是它的默认配置文件，而经典检索将读取遗留集合。
2. v2重新索引不会从一开始就删除遗留向量。
3. 虽然知识库不支持v2，但摄取将经典的原始条目写入遗留集合，并将完整的超集写入v2集合。
4. 每个成功的回调将其文档模式版本标记为2。
5. 当所有活动文档到达版本2时，检索将切换到每个概要文件的v2集合。
6. 该知识库的遗留向量在切换后被删除。通过带有显式遗留收集目标的现有矢量清理补偿机制记录失败的删除。

重新索引在作业排队之前重置受影响的文档架构版本。在部分迁移期间的新上传对classic仍然可见，因为Worker将它们的原始块双重写入到遗留集合。

##评估指标

每个评估结果记录了三个独立的结果：

- ‘ hitPassed ‘：返回的命中数满足案例的’ minHits ’。
- ‘ citationEligible ‘：案例有一个非空的’ expectedFileName ’。
- ' citationPassed '：一个符合条件的案例返回了期望的文件。

现有的总体“通过”结果继续包括点击、引用和预期令牌需求。

对于已完成运行的每个配置文件，计算：

- ‘ hitRate = hitPassedCount / caseCount ’
- ‘ citationRate = citationpasssedcount / citationEligibleCount ’
- ‘ casePassRate = passedCount / caseCount ’

‘ casePassRate ’是诊断性的，不控制默认启用的门。

## Gate Decision

门比较需要：

配置文件索引已经准备好了
-运行包括“经典”和候选人简介
-跑步是“完成”；
-运行版本与当前知识库版本匹配；
每个对比剖面至少存在3个案例
-每个比较的资料至少存在1个符合引用条件的案例。

考生只有在以下情况下才能通过考试：

-候选命中率大于或等于经典命中率和
-候选被引率大于等于经典被引率。

门的状态是：

——“通过”
——“阻塞”
——“INSUFFICIENT_DATA”
——“过期”
——“INDEX_NOT_READY”
——“NOT_EVALUATED”

运行的“gate_summary”存储运行中包含的每个非经典配置文件的基线指标、候选指标、速率增量、状态和机器可读原因。

默认配置文件强制执行

新的知识库必须从“经典”开始。指定其他默认值的创建请求将被拒绝。

更改默认配置文件遵循以下规则：

-切换到“经典”总是允许的
切换到非经典配置文件需要“维护”权限
- API加载最新完成的运行，包含经典和目标配置文件；
-存储的门必须是“通过”的，并且对于知识库的修订是当前的；
成功的切换只会更新默认配置文件，因为超集已经构建好了。
-一个失败的交换机返回HTTP 409与一个稳定的代码和一个具体的原因。

系统不会自动切换默认配置文件。

## API和Web界面

知识库响应增加：

——“indexSchemaVersion”
——“profileIndexReady”
——“indexRevision”
-每个非经典配置文件的最新gate状态

RAG评估运行响应添加：

——“indexRevision”
-每个配置文件指标
-每个候选人门的决定和delta

知识库设置面板显示每个候选概要文件的v2准备情况和最新门户状态。服务器保持权威；禁用或失效的客户端状态不能绕过该门。

RAG评估面板显示命中率、引用率、案例通过率、经典的delta和gate状态/原因。它不公开分数权重调优控制。

##故障处理

- QA生成失败仍然是非致命的。原始条目、父条目和子条目仍然完成摄取；依赖于qa的概要文件必须通过闸门证明质量。
-对知识库的非经典请求，其v2索引未准备好返回冲突错误，而不是静默搜索错误的索引。
—零命中的组合路由没有任何贡献，其他路由仍然返回结果。
- Milvus-wide failure保留现有的本地文本回退行为，PostgreSQL语料库通过请求的配置文件过滤。
-评估期间的修订变更标志着产生的门失效并阻止配置文件激活。
遗留清理失败不会回滚成功的v2切换；它进入补偿并保持可观察性。

安全与信任边界

现有的公共API认证、知识库范围检查和“维护”权限对于评估运行和配置文件变更仍然是强制性的。
-内部摄取和QA api在配置时仍然需要内部密钥。
-配置文件过滤器是从经过验证的枚举中派生出来的，永远不会从客户端请求中复制为任意的Milvus表达式。
- API根据持久化评估结果重新计算存储的门决策；Web客户端无法提交或覆盖门状态。

测试策略

# # #工人

—通用摄取生成具有正确作用域和连续块索引的五个条目组。
- QA失败仍然索引原始路由和子路由。
—传统的双写只发生在知识库没有准备好支持v2的时候。
-涵盖了Milvus v2模式创建、验证、插入、删除和配置文件过滤器。
—加权RRF是确定性的，组合处理空路由。
—混合语料库过滤排除请求配置文件之外的条目。

### API

V2集合模式和验证的配置文件表达式包含在“MilvusVectorServiceTest”中。
-矢量配置文件过滤和组合路由融合涵盖在“RetrievalServiceTest”。
-包括摄取完成、删除和重新索引修订更改。
-包括就绪切换和遗留清理补偿。
-门度量，最小数据规则，过时的修订，回归决策，和当前通过涵盖。
-知识库创建和配置文件更新实施在服务层、控制器层和授权层进行。

# # #网络

-知识库设置呈现就绪和当前/过时/阻塞/通过的门状态。
-配置文件选择不能将被阻止的候选显示为启用。
-评估比较呈现指标，delta和原因文本。
-保留现有的配置文件控制和RAG评估工作流。

最终验证

按照现有的Maven内存限制顺序运行API、Worker和以Web为重点的套件。运行Web生产构建，‘ git diff——check ’，检查生成的文件，并确认没有任务启动的Java、Node或Python进程。

## Rollout Rule

‘ CLASSIC ’仍然是唯一的无条件默认值。非经典配置文件只有在其当前版本门为“通过”后才可作为知识库默认配置文件进行选择。P2完成执行和诊断；它不授权全局自动配置文件更改。
