# V1.4.1上传治理设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-18-v1.4.1-upload-governance-design.md)





> **中文说明：** 本设计使文档上传与摄取在重试、取消、队列崩溃、过期回调和租户配额压力下保持安全，并将治理状态持久化且纳入审计。

# #目标

V1.4.1使文档的上传和摄取在重试、取消、队列崩溃、失效的工作者回调和租户配额压力下都是安全的。该版本保持了V1.4.0恢复保证的完整性，同时将上传治理从尽力而为的请求处理转变为持久的、可审计的状态。

##用户可见的结果

—用户可以在发送文件前查看剩余的上传配额。
-每个文件独立上传，具有有限的浏览器并发性和稳定的等幂键。
—失败或取消的文件不会毒害批处理的其余文件。
—取消排队的摄取作业将立即终止。
—取消正在运行的摄取作业请求取消，并且工作停止在安全检查点。
—重试的作业使用新的执行ID，因此忽略早期尝试的延迟回调。
—失败或死信摄取终端状态会发出单独的通知事件。
—取消用户不会发送失败通知。

现有系统的形状

- API是Spring Boot与PostgreSQL/Flyway， Redis队列，MinIO对象存储，和Milvus矢量清理。
- Worker是Python/FastAPI，消耗Redis，下载MinIO对象，解析/块/嵌入/索引，然后发布回调。
- Web是React/Vite，目前通过一个请求批量上传。
V1.4.0已经添加了恢复档案、恢复作业、稀疏恢复、接收发件箱、墓碑和本地发布扫描规则。

# #架构

# # # 1。持久租户上传配额

添加一个PostgreSQL配额分类账，有两种记账方式：

—保留配额：租户/用户当前拥有的持久字节数和文档数。
—上传窗口配额：在一个滚动时间窗口内可接受的字节数。

API在MinIO上传之前保留配额，只有在文档记录、对象键、摄取作业和摄取发件箱事件持久后才提交配额。如果MinIO或数据库工作失败，保留的配额将被释放。一旦API接受请求体，窗口字节不会被退还，因为这会保护上游带宽和本地资源。

配额使用显式保留：

—PENDING：保留容量，保留但尚未提交。
- COMMITTED：文件存在并保留使用计数。
—“已释放”：失败、删除、提交前取消后释放的容量。

每个上传请求可能包括一个幂等密钥。对于相同的承租者、用户、知识库和密钥，API返回原始文档/作业响应，而不是创建重复的文档。具有不同文件指纹的冲突密钥返回409。该实现还将' attemptId ' / ' attemptExpiresAt '存储在挂起的预订中，刷新跨MinIO/数据库边界的租约，并通过仅在文档、作业和发件箱持久时提交来协调过时的尝试。重新尝试已释放的幂等预约会重新检查保留的配额，但不会对滚动窗口字节进行双重收费。

恢复恢复作业绕过滚动上传窗口配额，因为它们不代表用户上传。恢复成功后，将保留的使用情况与恢复的文档字节进行比较。

# # # 2。可靠的摄取执行

Redis是一个传输层，而不是事实的来源。PostgreSQL拥有一个作业是否可以运行。

为摄取作业添加执行字段：

—executionId: Redis消息和回调中包含的当前尝试ID。
—callbackSequence：可接受的最高工作序列。
—claimedBy / leaseExpiresAt / starttedat / completedAt：索赔和租赁记账。
—cancelrequestdat：正在运行的取消状态。

Worker Redis消耗从破坏性的BRPOP变为准备处理的动作。Worker只有在PostgreSQL回调确认一个终端状态后才会移除处理项。如果Worker在移动消息后崩溃，Worker处理reaper扫描一个有界的最老的尾部，向API执行状态端点询问‘ terminal ’， ‘ leaseExpired ’和‘ requeueEligible ’，并自动将符合条件的有效负载移回ready。在国家敏感的工作之前，长时间的操作会使他们的租约生效。

声称规则:

—只有当status为PENDING且executionId与消息匹配时，才能声明排队的作业。
-索赔将作业移动到处理阶段，并使用租约进行解析。
—Retry旋转executionId，并将callbackSequence重置为0。
—恢复可以对租约到期的PENDING/QUEUED作业和stale PROCESSING作业进行重新排队。

# # # 3。取消

添加一个所有者/操作员可用的摄取取消端点。

—PENDING或排队作业：切换到CANCELLED，未提交时释放保留配额，取消发件箱事件，不排队。
—处理作业：转换为CANCEL_REQUESTED，并持久化cancelRequestedAt。
- Worker在解析，块，嵌入批处理，密集删除/索引，稀疏删除/索引以及最终完成回调之前检查取消。
-如果取消发生在索引开始后，Worker在发送CANCELLED之前只删除执行拥有的密集和稀疏向量。
—用户取消是终端，但不是失败，不会产生失败通知事件。

# # # 4。失效回调保护

每个回调都带有executionId和单调递增的序列。

API只接受当前作业executionId和序列大于存储的callbackSequence的回调。它还强制执行合法的向前转换：

- pending ->处理
-处理->完成
- processing ->失败
- processing ->已取消
- cancel_requested ->已取消
- FAILED -> retry创建新的executionId和PENDING
—DEAD_LETTER—> retry创建新的executionId和PENDING

终末状态永远不会因旧的回调而复活。重复的或过时的回调函数返回200，忽略=true，因此Worker重试不会放大失败。工作向量写也与每个文档的Redis锁和执行范围的确定性块id隔离；清理使用精确的‘ delete_by_ids ’进行密集和稀疏的存储，而不是文档范围的删除。

# # # 5。故障通知发件箱

为终端摄取失败创建一个独立的通知发件箱。它与摄取发件箱是分开的，因为摄取发件箱控制工作分派，并且不能重载用户通知语义。

规则:

-第一次转换到FAILED或DEAD_LETTER在同一事务中写入一个事件。
—事件关键字为jobId:executionId:status。
—重复终端回调不会重复事件。
—取消用户不会产生事件。
-交付重试使用有界指数回退并保留最后一个错误/状态。
通知webhook有效负载包括租户、知识库、文档、作业、executionId、状态、阶段和清理/截断错误。默认情况下，Delivery是HTTPS-only，除非明确允许，否则会阻止本地/元数据主机，并且可以包含‘ X-Dupi-Webhook-Secret ’。

# # # 6。Web上传管理

Web UI从一个批处理请求切换到每个文件编排。

—UploadZone接收配额和队列护栏数据。
-每个文件从名称，大小，修改时间和本地上传批ID生成一个幂等密钥。
-上传以有限的并发运行。
- AbortController只中止飞行中的HTTP上传；一旦文档/作业存在，用户取消调用摄取取消API。
- 409个幂等冲突，429个配额/窗口节流，413个文件过大的响应呈现每个文件消息。
—存在时显示“Retry-After”。
-轮询与AbortController序列化，因此陈旧的响应不能覆盖更新的状态。
—失败的通知通过jobId:executionId:status进行重复数据删除。

## API合约

—GET /api/v1/upload-quota
-返回保留的字节/文档使用率，配置的限制，滚动窗口使用率，以及下次重试时间（如果节流）。
—POST /api/v1/知识库/{kbId}/documents
-接受可选的幂等密钥头。
-返回DocumentResponse和currentJob。
-对于幂等键冲突返回409。
-如果配额/窗口耗尽，返回429，并使用Retry-After（如果适用）。
—如果文件大小超过配置的保留配额或multipart max，则返回413。
—POST /api/v1/知识库/{kbId}/ingest-jobs/{jobId}/cancel
-返回当前IngestJobResponse。
- POST /api/v1/internal/ingest/jobs/{jobId}/claim
-工人在工作前要求执行。
—POST /api/v1/internal/ingest/jobs/{jobId}/lease
- Worker刷新租约。
—POST /api/v1/internal/ingest/status
-接受executionId和sequence。
—返回{status:“ok“，忽略：false}或{status:”ignored“，忽略：true， reason:”…”}。
—GET /api/v1/internal/ingest/jobs/{jobId}/executions/{executionId}/state
-返回status， executionCurrent, terminal, leaseExpired，和requeeeligible用于Worker心跳和处理列表获取。

##测试策略

—API单元测试包括配额保留、幂等冲突、删除释放、取消转换、重试时执行轮换、忽略失效回调、终端回调重复数据删除和通知发件箱创建。
- API控制器测试包括上传配额端点、幂等密钥、409/429/413响应映射、取消端点和回调忽略响应。
- Worker测试包括准备处理移动、处理ACK、失败回调保留处理项直到确认、取消检查点、索引后取消时的清理、executionId/sequence回调有效负载和失效回调忽略处理。
Web测试涵盖每个文件上传排序、幂等密钥生成、中止与摄取取消的区分、配额显示、重试后显示和序列化轮询。
-版本检查使用JaCoCo， Worker pytest， Web测试/构建，Pester版本扫描，docker组合配置，diff检查和本地发布扫描运行API验证。

# #自我回顾

-占位扫描是干净的。
-范围故意是V1.4.1，不包括新的V1.4.2功能。
-该计划可在垂直切片中构建：持久性、API状态机、Worker传输、Web体验、通知传递、发布。
-设计风险集中在事务性配额+外部MinIO写；保留状态机使故障可审计且可逆。
用户体验风险集中在HTTP中止和摄取取消之间的差异上该计划明确指出了这种区别。
