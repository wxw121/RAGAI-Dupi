# V1.2.1端到端隔离和清理设计

<!-- language-switch -->
[English](../../../en/superpowers/specs/2026-07-14-v1.2.1-e2e-isolation-cleanup-design.md)





> **中文说明：** 本设计把真实浏览器回归数据隔离到 `e2e` 租户，并在成功运行后删除测试资源；失败时仅保留诊断所需的数据和标识符。

# #目标

在“默认”租户中保留重复的真实浏览器回归运行，并删除所有成功运行的测试数据。当运行失败时，只保留相关的‘ e2e ’租户数据和诊断它所需的标识符。

# #范围

-添加一个严格限制的API，用于删除‘ e2e_* ’测试帐户。
-在短暂的“端到端”租户管理员会话中运行browser-gate业务流。
—自动移除已成功运行的知识库和临时帐户。
-在‘ e2e ’租户中保留运行失败的证据，并在剧作家输出中公开其标识符。

该版本没有向Web帐户管理页面添加一般的帐户删除控制，没有删除审计记录，也没有自动删除先前的历史测试数据。

帐户删除边界

添加“DELETE /api/v1/ops/accounts/{username}”。

现有的‘ /api/v1/ops/** ’过滤器需要‘ OPS_ADMIN ’。‘ AccountService ’在删除之前增加了第二个不可绕过的策略检查：

1. 用户名必须以小写字母“e2e_”开头。
2. 帐户租户必须完全为“e2e”。

所有其他请求返回验证错误，包括‘ default ’或任何普通帐户中的‘ e2e_* ’名称。成功后，帐户将从‘ user_accounts ’中物理移除，并且服务记录‘ ACCOUNT_DELETE_E2E ’对应目标类型‘ account ’。审计日志保留，保留删除事件，而不留下可登录的帐户。

故意不将API添加到常规Web帐户UI中。它是一个测试基础结构控件，只能从自动化或管理员操作的恢复命令中调用。

浏览器闸门生命周期

每次运行都会创建一个‘ runId ’，并派生出以下标识符：

-管理员：‘ e2e_gate_<runId> ’在租户‘ e2e ’中，角色‘ ADMIN ’
-验证帐户：‘ e2e_account_<runId> ’在租户‘ e2e ’
-知识库：‘ e2e-browser-<runId> ’在租户‘ e2e ’

已配置的管理员先登录仅用于创建‘ e2e_gate_<runId> ’。它不会在“默认”中创建知识库和测试文档。然后，测试以“e2e_gate_<runId>”身份登录，并执行现有的混合知识库、文档/聊天/RAG-eval、审计、帐户和角色流。

该测试在剧作家输出中记录了三个标识符。只有在每个产品断言和浏览器错误门通过后才设置完成标志。在“finally”块中：

—运行成功：删除知识库，删除e2e_account_<runId> ‘，再通过受限API删除e2e_gate_<runId> ’；断言已清理的资源不再出现。
-运行失败：跳过清理并将标识符和当前URL附加到测试输出，以便在租户‘ e2e ’下仍然可以检查证据。
—cleanup failure：运行失败，挂载剩余资源。它不能静默地报告一个成功的门。

清理HTTP调用使用经过身份验证的浏览器会话和与Web控制台相同的CSRF机制。这使测试服从于真正的授权和CSRF边界，同时避免了额外的UI特性，仅用于测试分解。

# #测试

后端测试包括成功删除和拒绝非‘ e2e_* ’用户名或非‘ e2e ’租户，以及‘ ACCOUNT_DELETE_E2E ’审计记录。控制器测试覆盖端点委托，现有的“OPS_ADMIN”过滤器仍然是外部权限门。

Web API测试涵盖了DELETE客户端助手。真正的Chromium gate证明了完整的成功路径：它必须创建端到端租户资源，执行现有的浏览器检查，删除所有三个资源，并验证配置的管理员的“默认”知识库列表没有获得运行的知识库。故意失败的运行不是正常闸的一部分；故障分支由单元级测试工具断言和剧作家保留的跟踪/截图/视频配置覆盖。

##操作说明

在此版本之前失败的运行将‘ e2e-browser-* ’ data留在‘ default ’；它们在自动清理范围之外。管理员可以分别检查和删除它们。在V1.2.1之后，正常大门成功只留下可审计的“e2e”租户审计记录，没有测试知识库或用户帐户。
