> **中文说明：** 本手册规定 V1.5.0 RAG 质量升级的兼容性要求、发布前检查、部署步骤、质量门禁、回滚方案和最终发布证据。

**状态：** 释放。只有在本文档中的每个最终闸门通过发布提交并且合并的“主”CI为绿色之后，才创建“v1.5.0”标签。

# #范围

V1.5.0增加了知识库级别的`CLASSIC`， `PARENT_CHILD`， `QA_ASSISTED`和`COMBINED`检索配置文件，而不替换现有的`VECTOR`， `HYBRID`和`RERANK`执行模式。Worker摄取构建一个可过滤的Profile V2超集，其中包含原始条目、父条目、子条目和QA条目。只有在当前索引修订通过候选与经典质量检验之后，非经典配置文件才能成为知识库默认值。

# #兼容性

- 没有选择检索配置文件的现有请求继续使用“CLASSIC”。
- ' VECTOR ', `HYBRID`和可选的重新排序保留其现有的API行为。
QA生成失败不影响原始文档的摄取。
- 配置文件选择改变检索入口路由；它不会重新构建已经存在的配置文件V2索引。
- 恢复后的知识库通过V1.4恢复合同保留其配置文件收集数据、门状态和割接状态。

# #起飞前的

1. 备份Flyway V18-V20之前的PostgreSQL、MinIO等和Milvus。V18增加了检索索引模式，V19增加了Profile V2修订/准备和评估证据，V20保留了Profile V2切换状态。
2. 记录每个知识库的文档计数、已完成文档计数、当前默认配置文件、活动检索配置文件、索引修订和当前Milvus集合计数。
3. 为API和Worker配置相同的非空`MILVUS_PROFILE_COLLECTION`。它不能重用`MILVUS_COLLECTION`或其他环境的集合。
4. 确认Milvus在一次性迁移期间具有原始/父/子/QA超集的容量，并在切换之前具有临时Legacy双写的容量。
5. 对于最初的rollout，保持`COMBINED_CHILD_WEIGHT=1.0`和`COMBINED_QA_WEIGHT=0.8`，除非保留的基准证明了不同的值。
6. 为API和Worker使用相同的非空`DUPI_INTERNAL_KEY`。确认用于QA生成的LLM端点是可达的，但不要仅仅因为QA生成不可用而阻止rollout。
7. 保留“CLASSIC”的所有现有知识库，直到其配置文件V2索引准备就绪，并且其当前版本通过了质量检查。

# #推出

1. 首先部署PostgreSQL/Flyway和API。在启动Worker之前，请验证`/actuator/health`、Flyway V18-V20和现有知识库列表。
2. 使用`MILVUS_PROFILE_COLLECTION`、组合权重和现有可靠队列设置部署Worker。新的摄取双写入遗留原始向量，直到知识库达到配置文件V2切换。
3. 在API和Worker运行正常后部署Web。在迁移数据之前，验证知识库设置和RAG评估面板的加载。
4. 选择一个非关键知识库并运行一个索引。重新索引卷文件到位；不要先删除Profile V2集合或清除所有活动块。
5. 等待，直到每个文档`COMPLETED`与`index_schema_version=2`。在割接清理之前，确认配置文件V2准备就绪，非空索引修订，以及预期的原始/父/子/QA计数。
6. 为“CLASSIC”和每个候选人的简介运行至少三个可引用的可评估案例。候选人不得回归点击率或引用通过率，结果必须参考当前索引修订。
7. 只有当非经典默认值的决定是`PASSED`时才激活它。将`BLOCKED`， `INSUFFICIENT_DATA`， `STALE`， `INDEX_NOT_READY`和`not_evaluate`视为硬停止；API返回`409 retrieval_profile_gate_blocked`表示被阻止的激活。
8. 每次扩展一个知识库。以`PARENT_CHILD`开头；只有在保留的评估证据显示有好处时，才启用“QA_ASSISTED”或“COMBINED”。

# #监控

- 在`index_schema_version=2`以下的文档，过时的索引修订，除`PASSED`之外的gate决策，配置文件V2清理失败以及重新索引期间的摄取失败时发出警报。
- 跟踪配置文件V2收集增长、QA条目计数、QA生成失败、检索回退原因、重新排名回退、命中率、引用通过率和每个配置文件的P95延迟。
- 保留发布扫描、金丝雀评估运行id、索引修订、收集计数和回滚结果，删除秘密。

# #回滚

1. 将知识库的默认设置切换回“CLASSIC”；这是无条件的，不需要候选人门的证据。
2. 在调查候选质量或延迟回归时，不要删除Profile V2集合。它是所有四个概要文件和可重复评估的共享源。
3. 如果在概要文件V2切换之前Worker rollout失败，停止Worker，保留可靠的处理队列，恢复释放Worker，并从持久化文档/索引状态恢复。
4. 首选API/数据库故障的前向修复。V18-V20是可添加的，但手动更改`profile_index_activated`， `index_revision`或文档模式版本可以重新启用无效的回退路径。
5. 在遗留清理之后，回滚到只理解遗留集合的代码需要经过验证的存档/还原或完整的重新索引。不要清除持续切换状态以强制回退。
6. 首先回滚Web，然后回滚Worker，只有在确认旧组件理解当前队列有效负载和数据库模式之后再回滚API。
