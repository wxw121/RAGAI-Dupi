## 发布门槛

V1.3 发布必须同时满足：30 条检索用例清单校验通过，VECTOR/HYBRID/HYBRID+RERANK 均未被质量门禁阻断，Sparse 检索无 fallback，升级备份清单完整，供应链扫描无未修复 CRITICAL 和禁止许可证，Worker 镜像不超过 3 GB。

仓库提供了自动化和策略单测，但生产同规格演练、真实环境 30 Case 基准、完整 CVE/许可证/镜像扫描仍必须在发布环境执行并保存产物。不能用仓库单测结果替代环境证据。

## 30 条检索基准

清单位于 `benchmarks/v1.3/retrieval-cases.json`，中文、英文、精确关键词、语义改写、无答案和冲突文档各 5 条。配套当前事实与 legacy 冲突文档位于 `benchmarks/v1.3/corpus/`。`caseKey` 是稳定身份；同步脚本按文件名上传缺失语料并等待摄入完成，只创建或更新清单内用例，不删除额外的运维用例。

```powershell

powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-eval-cases.ps1 `
  -KbId <kb-id> -BaseUrl http://localhost:8080 -ApiKey <api-key>

powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-retrieval-benchmark.ps1 `
  -KbId <kb-id> -HybridProfileId <profile-id> -RerankProfileId <profile-id> `
  -BaseUrl http://localhost:8080 -ApiKey <api-key>

```

报告写入 `artifacts/`，包含一次冷运行、三次热运行、分类覆盖、Profile 快照、质量指标、P50/P95、fallback 和逐 Case 排名证据。少于 30 条、缺分类、`BLOCKED`、fallback 或缺失 Rerank 排名会直接失败。无答案 Case 使用 `minHits=0` 且没有正向文件/关键词断言，语义是严格零命中；若检索器不能拒答，质量门禁应失败。

## Rerank 预热与缓存

Worker 启动时默认加载 `BAAI/bge-reranker-base` 并执行一次小推理。以下变量可配置：

- `RERANK_MODEL`：模型名。
- `RERANK_WARMUP_ENABLED`：是否在启动生命周期预热，默认 `true`。
- `HF_HOME`、`SENTENCE_TRANSFORMERS_HOME`、`TRANSFORMERS_CACHE`：Compose 中统一为 `/models/huggingface`。

命名卷 `hf_model_cache` 保留模型权重。`GET /health` 返回 `rerankerConfigured`、`rerankerReady` 和 `rerankerModel`，不返回本地缓存路径或异常详情。预热失败不会使 VECTOR/HYBRID 停止服务，但 Rerank 请求会带 `reranker_unavailable` fallback 证据。

发布目标是同规格环境热态 HYBRID+RERANK P95 不高于 250 ms，且质量不低于当前模型。没有真实硬件基准前不自动切换更小模型。

## Milvus 2.4.1 到 2.5.4 演练

演练脚本默认使用隔离 Compose project，并在任何升级动作前备份 PostgreSQL、etcd、MinIO 和 Milvus 数据卷。输出清单记录版本、阶段、耗时、文件大小和 SHA-256。

```powershell

$env:POSTGRES_PASSWORD = '<secret>'
$env:MINIO_ACCESS_KEY = '<access-key>'
$env:MINIO_SECRET_KEY = '<secret-key>'
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rehearse-v1.3-upgrade.ps1 `
  -Environment rehearsal -ProjectName dupi-v13-rehearsal `
  -KbId <kb-id> -HybridProfileId <profile-id> -RerankProfileId <profile-id> `
  -RollbackProfileId <previous-pass-profile-id>

```

生产目标必须额外传入 `-AcknowledgeProduction`。脚本拒绝把备份写入 `deploy/` 数据路径，要求至少 5 GB 空间，并在备份少于四类或任一命令失败时停止。成功清单的 `backupVerified` 必须为 `true`。

Sparse Migration 在知识库 RAG 评估页操作：`PREPARING -> BACKFILLING -> DUAL_WRITING -> SHADOW_VALIDATING -> CUTOVER -> COMPLETED`。Cutover 要求覆盖率 100%、维度一致、候选 P95 不超过基线 1.25 倍、fallback 不增加，且 API 再次校验质量门禁。Legacy BM25 fallback 只允许在 `DUAL_WRITING` 和 `SHADOW_VALIDATING` 使用。

Rollback 不是修改历史数据，而是重新激活已有匹配 `PASS` 评估的旧 Profile。Cutover 后必须执行一次 Rollback 和检索验证，再恢复候选 Profile。

## 供应链扫描

本地需要 `pip-audit`、Syft、Trivy 和 Docker：

```powershell

powershell -NoProfile -ExecutionPolicy Bypass -File scripts/scan-release.ps1

```

扫描输出位于 `artifacts/v1.3-release-scan/`，包含 Python audit、Syft/CycloneDX SBOM、Trivy JSON 和 Markdown 摘要。`deploy/license-denylist.txt` 默认禁止 AGPL-3.0、GPL-3.0 和 SSPL-1.0。未修复 CRITICAL、禁止许可证、超过 3 GB 会阻断；HIGH 必须提供带日期的例外文件。

cd ../web; npm test; npm run build; npx playwright test
