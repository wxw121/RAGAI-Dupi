> **中文说明：** 本手册介绍 V1.4.1 上传治理版本的发布前检查、上线步骤、验证与回滚要求，覆盖配额、幂等、任务取消、过期回调防护及失败通知。

# #范围

V1.4.1增加了持久的上传配额保留、每个文件的幂等性、可靠的摄取执行/ACK行为、取消、陈旧的回调保护、终端故障webhook交付和受管理的Web上传。V1.4.0恢复档案保持不变。

# #起飞前的

1. 备份Flyway V17之前的PostgreSQL。V17创建了‘ upload_quota_reservations ’， ‘ upload_window_events ’和‘ ingest_failure_notifications ’，添加了‘ documents ’。Quota_reservation_id ‘，并将执行/租赁列添加到’ ingest_jobs '。
2. 确认Redis具有就绪队列和‘ INGEST_PROCESSING_QUEUE ’的容量。
3. 配置README中记录的配额变量。默认值为1 GiB保留字节、1,000个保留文档和每个租户/用户每个一小时滚动窗口256 MiB。保留使用计数活动的“PENDING + COMMITTED”预留；‘已释放’的行不消耗配额。
4. 如果操作需要外部故障警报，配置‘ INGEST_FAILURE_NOTIFICATION_WEBHOOK_URL ’；当它是空白时，传递将被跳过。为目标端点调整‘ INGEST_FAILURE_NOTIFICATION_TIMEOUT_SECONDS ’， ‘ INGEST_FAILURE_NOTIFICATION_MAX_ATTEMPTS ’和‘ INGEST_FAILURE_NOTIFICATION_DISPATCH_CRON ’。
5. 为共享环境中的API和Worker使用相同的非空‘ DUPI_INTERNAL_KEY ’。
6. 在rollout期间不要删除处理列表。这是就业复苏的证据，表明就业岗位已从准备就绪名单中移出，但尚未得到最终承认。

# #推出

1. 首先部署PostgreSQL/Flyway和API。
2. 使用具有“ DOCUMENT_UPLOAD ”的帐户验证“ GET /actuator/health ”和“ GET /api/v1/upload-quota ”。
3. 使用‘ INGEST_PROCESSING_QUEUE ’， ‘ INGEST_LEASE_SECONDS ’和唯一的‘ WORKER_ID ’部署Worker。
4. 在API和Worker运行正常后部署Web。
5. 上传一个带有“幂等密钥”的小文档；重播相同的请求并确认返回相同的文档/当前作业。
6. 重复使用不同内容的密钥并确认409。在413中使用过大的文件，在429中使用耗尽的窗口，再加上“Retry-After”。
7. 取消一个排队作业和一个正在运行的作业。确认队列变为‘ CANCELLED ’；运行从‘ CANCEL_REQUESTED ’移动到‘ CANCELLED ’，没有失败通知。
8. 重试失败的作业并确认‘ executionId ’更改。发送一个旧执行或重复序列回调，并使用‘ ignore =true ’确认HTTP 200。
9. 当配置了‘ INGEST_FAILURE_NOTIFICATION_WEBHOOK_URL ’时，强制一个FAILED的摄取并确认一个webhook交付；重复的终端回调不得创建另一个事件。一个失败的webhook应该通过‘ FAILED ’重试事件，最后是‘ EXHAUSTED ’。

##操作检查

-将“retainedBytesUsed”和“retainedDocumentsUsed”与活动的“PENDING + COMMITTED”保留和实时文档进行比较；在手动更改配额总数之前，在对账中包括旧的待处理预订。
-在手动更改配额总数之前，调查旧的‘ PENDING ’预订和旧的处理列表消息。
- ‘ CANCEL_REQUESTED ‘作业应该保持轮询，直到Worker返回’ CANCELLED ’。
——“ingest_failure_notifications。Event_key '是唯一的。重复的回调不能创建另一行；从‘ PENDING ‘ / ‘ FAILED ‘行重新尝试webhook交付，直到’ DELIVERED ’或’ EXHAUSTED ’。
-恢复恢复应为恢复的文件创建“COMMITTED”保留；放弃恢复应该释放关联的保留，理由是“恢复恢复已放弃”。
—Web transport abort仅停止当前HTTP上传。如果响应已经创建了文档/作业，则使用摄取取消端点。

# #回滚

1. 首先回滚Web；批处理API仍然可用以保持兼容性。
2. 在回滚API之前停止Worker，这样V1.4.1索赔/租赁/回调合约就不会被发送到旧的API。
3. 更喜欢API/数据库问题的前向修复。V17是附加的，但是删除它的表/列会破坏配额和执行证据。
4. 如果Worker rollout失败，保留处理列表，恢复V1.4.1 Worker，并让PostgreSQL执行/租赁状态决定是否可以恢复工作。
